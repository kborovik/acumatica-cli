## T29: financial currency joins the Bootstrap endpoint — and diff learns a second read

B8 left the last SalesDemo-replay wall standing: the Default endpoint's
`Currency` entity is the CM201000 currency *list* only, so a fresh tenant

and the seed data must supply real accounts. Value types follow the
Default endpoint's own naming (`ShortValue` for the Int16, `BooleanValue`
for the two flags, `StringValue` for segment-mask CD strings — the
swagger dump's component names and the T12 import round-trip agree).

The live verify ran the whole pipeline on a fresh scratch tenant:
provision (the new package published through the T25 digest gate without

endpoint, then `PUT Currency [EUR]` through `Bootstrap/1.0.0`, then the

## B9/T30: the list GET has an optimizer, and delegate views break it

The verify's diff leg found the next wall immediately: `acu diff` on the
currencies YAML exited 1 with a 500 — "Optimization cannot be performed
... View CuryRecords has BQL delegate", one line per GL-account field.
The contract API's *list* GET (the `$filter` idiom diff has always used)
runs an optimized export that hard-fails when any field in scope maps to
a delegate-backed view; `$select` narrowing does not help once such a
field is named, and the default scope is all fields. The probe that
mattered: a *single-record* GET by key URL
(`/entity/Bootstrap/1.0.0/Currency/EUR`) skips the optimizer entirely
and returns the full record, GL accounts included. So the PUT persists,
the EUR account proves it — but diff could not read it back, and
provision diffs everything it applies, so a currencies file would have
poisoned the whole pipeline.

Backpropped as B9 with the recurrence class spelled out (any
custom-endpoint entity mapping a delegate view is unreadable via list
GET — the diff idiom's hidden assumption), and the operator picked the
fallback remedy: `$filter` list GET stays the primary read, and exactly
the optimization-500 reroutes that record to a new
`client.get_record` key-URL GET. One more probe pinned the missing-record
signature on the key-URL form — a 500 carrying
`NoEntitySatisfiesTheConditionException`, not a 404 and not an empty
list — which the fallback maps to the familiar "missing on tenant"
drift line. Offline tests pin the fallback round-trip, the
fallback-missing path, and that any *other* 500 still raises; live,
the currencies diff went from 500 to a clean exit 0, and the scratch
tenant came down with its recycle.

The baseline expansion order (currencies first) now has its mechanism
end-to-end: the endpoint exposes the entity, the data repo can carry a
`currencies.yaml` (in `baseline/` with the `endpoint:` override, so
accounts sort and seed ahead of it), and diff can verify it.

## Code review over src/: two fixes, two preventive spec rows

A high-effort multi-agent review of `src/` after T29 surfaced four
findings that survived adversarial verification (two more were refuted).
Two were fixed on the spot:

- The template comment in `bootstrap_project.xml` told seed authors the
  ten gain/loss pairs are "required at persist, so the seed data must
  supply them" — the exact opposite of the live-verified `Translation*`
  caveat (friction #37: write-tolerated, read-invisible while
  financial-statement translation is off), which until now lived only in
  the T29 commit message. Anyone following the comment would have built
  seed YAML that diffs dirty forever. The caveat is now in the comment
  itself. Worth remembering the failure shape: a live-verified fact
  recorded in a commit message but not in the artifact it constrains is
  invisible at the moment it matters. Side effect: the comment edit
  changes the package digest, so the next publish against a live tenant
  will republish once.
- `PACKAGE_DESCRIPTION` in `bootstrap.py` was a hand-synced duplicate of
  the XML root `description` attribute — the T29 diff had to edit the
  same sentence in two files, and a divergence would have either desynced
  the tenant-visible description or fired a spurious digest-gate
  republish. The constant now parses the shipped template; the XML is
  the single source.

The other two findings were latent design gaps, not observed failures,
so they went into the spec preventively (operator's call: amend, not
backprop — the §B ledger stays a record of things that actually broke):

- **V20 + T32** — the Bootstrap endpoint's `Currency` entity shares its
  name with the Default endpoint's CM201000 currency-list entity, and
  the per-file `endpoint:` override is optional; a `currencies.yaml`
  that forgets the override would PUT to the wrong screen, report
  success, and bring B8's 422 back behind a clean apply. T32 makes
  `seed.py` hard-error on any entity the shipped Bootstrap template
  serves when the file names no endpoint.
- **V21 + T33** — T29 changed the endpoint's contract shape but kept
  `Bootstrap` at version 1.0.0, so two different contracts share one
  name-plus-version identity; an older acu build touching an
  already-bootstrapped tenant would see a digest mismatch and "repair"
  it by republishing the currency-less contract under the same name — a
  silent downgrade with no version signal in the resulting seed
  failures. T33 bumps to 1.1.0 and lists the ripple targets (templates,
  docstring example, test fixtures, data-repo refs).

## T32: the ambiguous-entity guard lands

The V20 gap the code review specced preventively is now enforced:
`load_baseline` hard-errors on any file whose `entity` the packaged
Bootstrap endpoint serves when the file names no `endpoint:` — the
error names both endpoints (the instance default and the packaged

second hardcoded copy. Offline tests pin the parse parity against the
shipped template, the ambiguous-file bail, and that an explicit
`endpoint:` in either direction still loads. Purely offline work — no
live verification needed, the guard fires before any HTTP.

## T38: the action-file mechanism lands (offline)

The design from `designs/setup-action-files.md` folded into the spec this
morning became code this evening. The seeding layer now understands a
second file kind: a `setup/*.yaml` action file declares a contract action
(`action` + `entity` + `record`, optional `parameters` and `endpoint`)
plus a `done_when` live-state probe, and `load_baseline` turned into a
dispatcher on the `action:` key returning `BaselineFile | ActionFile` —
`apply`, `diff`, and `provision` absorb the kind transparently, and
`provision` gained the `setup/` phase between `baseline/` and the drift
check. The V20 ambiguity guard now covers both kinds: an action file
naming a Bootstrap-template entity without an explicit `endpoint:` gets
the same hard error a baseline file does.

Client-side, `invoke()` POSTs `/entity/<endpoint>/<Entity>/<Action>` with
both payloads value-wrapped. The status-code contract encodes the trap
T36's live archaeology hit: a 204 means done *and can still carry a
`Location` header that resolves nowhere* — only a 202 means poll. The
poll loop resolves `Location` per HTTP semantics against the request URL
(joining, not string-prefixing, so an instance-prefixed absolute path
does not double up against the client's `base_url`) and re-polls the
same URL until the server answers 204.

Idempotence stayed V4-shaped on both sides of the probe: `apply` runs
`done_when` first and emits `skip <action> (already done)` on a non-empty
result; `diff` runs the same probe and emits `action <name>: not applied`
(exit 2) on an empty one. One probe, both directions, coarse
present/absent — an action leaves no keyed record to field-diff, and a
marker would outlive state loss. Dry-run emits `would invoke <action>`
and makes zero HTTP calls, matching the PUT path's discipline.

Twelve offline tests pin the mechanism through the real client over
`httpx.MockTransport`: dispatcher parse and V10 rejection, the required
`done_when`, skip-on-probe, the 204-with-bogus-Location sequence (the
request list proves the Location was never followed), the 202-poll to
completion, payload wrapping, dry-run, both diff directions, and probe
routing (entity/endpoint defaulting to the action's own, `filter` riding
`$filter`).

One friction worth recording: the working tree carried T36's in-flight
`bootstrap_project.xml` (already bumped to 1.3.0), which fails three
version-pinned offline tests and would trip the V21 parity sweep. T38
neither needs nor may bundle it (write-ownership), so it sat in a stash
across the verify gate and the path-scoped commit, restored after. Same
current-version discipline applied inside T38's own artifacts: the test
fixtures and the seed.py docstring example cite `Bootstrap/1.2.0` today,
and T36's bump sweeps them with the rest of the V21 ripple.

The version ripple was T33's machinery again — `BOOTSTRAP_ENDPOINT`,

plus one new wrinkle: the endpoint parity test had assumed every
mapping's To-field equals the entity field name, and the calendar

First live verify: four of five legs green, and the fifth was a fresh
bug class. Both actions worked end to end on a fresh tenant —
`GeneratePeriods` realized the FinYearSetup singleton, `GenerateCalendar`
generated 2026, and the `done_when` probe on CompanyCalendar answered
non-empty, which also settled the open question of whether company

LAB5]: missing on tenant`, behind a PUT that answered 200 twice — B12's
silhouette, so the reflex said dropped write. The autopsy said
otherwise: the `OrganizationLedgerLink` row was in SQL, the unfiltered

precisely in the filter conjunction. `LedgerCD eq 'ACTUAL'` alone

empty list. The key-URL form 500s "Operation is not valid" — a

unreadable-back: the write succeeds, the diff reports missing, forever.

files key on primary-view fields only; secondary-view fields stay

`key: LedgerCD`. Second e2e run: five of five, fresh tenant, zero
manual steps, both action skips asserted on the re-run leg.

Note the asymmetry that made the diagnosis fast: tenant deletion had

write-proof for the fresh-tenant case then rode the second e2e run

diff is the proof the PUT creates one.

GL posting now has exactly one link left: the generated periods land
Inactive, and opening them is T37's processing-screen problem (B13).

## T37: the processing screen falls — GL posting closes end to end

B13 left GL posting one link short: generated periods land Inactive, and
the screen-side "Open Periods" button is a `PXRedirectHelper` redirect to
GL503000 that the contract API cannot follow. T37's question was what CAN
drive GL503000, and the answer turned out to be the contract API itself —
just aimed at the processing screen directly instead of at the redirect.
`ManagePeriods` (Bootstrap 1.4.0) maps the `FinPeriodStatusProcess`
graph's `Filter` view — Action, FromYear, ToYear, OrganizationID — and
carries a contract action mapped to `ProcessAll`, which
`PXFilteredProcessing` registers at runtime (no declared `PXAction`
member anywhere on the graph; static reflection shows only
`ShowDocuments`). Setting the filter through the entity record and
invoking ProcessAll processes every period the filter matches, no per-row
`Selected` flags involved.

The archaeology instruments were T36's, in the same order: the aspx grep
settled the graph, primary view, and field bindings in one pass, and
constructor-free reflection pinned the stored string values — the filter
`Action` takes the literal words `Open|Close|Lock|Deactivate|Reopen|
Unlock` (the ten-character `Deactivate` explains the field's length), and
period `Status` stores `Inactive|Open|Closed|Locked`, words not letters.
One instrument note for the file: a UTF-16 string-heap scan must try both
byte alignments — the first pass declared `ProcessAll` absent from
PX.Data.dll because the literal sat at an odd offset.

The done_when probe got its own entity. CompanyCalendar's year rows carry
no status, so `CompanyPeriod` fronts GL201100's `OrgFinPeriods` view —
FinPeriodID, FinancialYear (the DAC's `FinYear`), Status — with every
field deliberately on that one view: B14 taught that a filter conjunction
spanning views answers 200 with an empty list, and this probe ANDs
FinancialYear with Status. The dead CompanyCalendar `OpenPeriods` action
(shipped in 1.3.0 as T37 probe material) left the contract in the same
bump.

Live verify ran the full chain on a fresh scratch tenant: provision with
zero manual steps (the digest gate saw the changed package and
republished), all twelve files applied, `invoke ProcessAll
[ManagePeriods]` ran, and the closing diff read clean — which is itself
the probe's proof, since the diff leg runs the same done_when. Then the
leg three tasks kept deferring: a GL batch PUT through the Default
endpoint (hand-built payload — `wrap()` is scalar-only and Details is a
list, so the probe script talks to the client's session directly),
released via `ReleaseJournalTransaction`, and the batch came back
**Posted** — B12's "07-2026 does not exist", T34's 422s, and B13's
inactive-period wall all gone in one pass. Re-run apply skipped all three
actions, the re-diff stayed clean, and the tenant was deleted with the
list verified back to four.

Two loose ends surfaced and stayed out of scope per the task boundary:
the `config init` template set still ships no `setup/` files (a
scaffolded tenant gets no calendar and no open periods, so T35's
batch-posting leg only closes through the data repo, not the templates),
and the §V.21 parity recipe in check-extras scans `baseline/` +
`bootstrap/` but not the newer `setup/` symlink. Both belong to the
operator's next dispatch, not this build.
