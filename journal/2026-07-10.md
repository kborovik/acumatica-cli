# 2026-07-10

## T29: financial currency joins the Bootstrap endpoint — and diff learns a second read

B8 left the last SalesDemo-replay wall standing: the Default endpoint's
`Currency` entity is the CM201000 currency *list* only, so a fresh tenant
has no CM.Currency (financial-currency) rows and every EUR-denominated
account PUT answers 422 "Currency 'EUR' cannot be found". T29 fronts
CM202000 from the custom Bootstrap endpoint, the same route that closed
the company and credit-terms gaps.

The archaeology (all via `scripts/ps-remote`, the reflex this time —
friction #33 not repeated): CM202000 resolves to `CurrencyMaint` with
primary view `CuryListRecords` over the *list* DAC (`CM.CurrencyList`) —
the general-info form binds `CuryID`, `Description`, `CurySymbol`,
`DecimalPlaces`, `IsActive`, and crucially `IsFinancial` there, while the
GL Accounts tab binds to `CuryRecords` (`CM.Currency`, Current-keyed on
the list row). DAC reflection showed every gain/loss `AcctID`/`SubID`
pair on CM.Currency carrying `PXDefault(PersistingCheck.Null)` with no
setup default, which settles the entity's field set: all ten pairs ship,
and the seed data must supply real accounts. Value types follow the
Default endpoint's own naming (`ShortValue` for the Int16, `BooleanValue`
for the two flags, `StringValue` for segment-mask CD strings — the
swagger dump's component names and the T12 import round-trip agree).

The live verify ran the whole pipeline on a fresh scratch tenant:
provision (the new package published through the T25 digest gate without
any manual invalidation — the content-aware skip earning its keep), then
subaccount 000 + gain/loss accounts 8710/8720 through the Default
endpoint, then `PUT Currency [EUR]` through `Bootstrap/1.0.0`, then the
B8 acceptance object itself — an EUR-denominated account — applying and
diffing clean. Two wrinkles worth recording: the editable-install marker
from T28 confirmed at a glance that the running provision was building
the package from the just-edited source tree; and the `Translation*`
account pairs turned out to be feature-gated — with financial-statement
translation off, the API accepts them in a PUT but never returns them,
so seed YAML on this feature set must not claim them (they would diff
"not returned by endpoint" forever).

## B9/T30: the list GET has an optimizer, and delegate views break it

The verify's diff leg found the next wall immediately: `acu diff` on the
currencies YAML exited 1 with a 500 — "Optimization cannot be performed
... View CuryRecords has BQL delegate", one line per GL-account field.
The contract API's *list* GET (the `$filter` idiom diff has always used)
runs an optimized export that hard-fails when any field in scope maps to
a delegate-backed view; `$select` narrowing does not help once such a
field is named, and the default scope is all fields. The probe that
mattered: a *single-record* GET by key URL
(`/entity/Bootstrap/1.0.0/Currency/EUR`) skips the optimizer entirely
and returns the full record, GL accounts included. So the PUT persists,
the EUR account proves it — but diff could not read it back, and
provision diffs everything it applies, so a currencies file would have
poisoned the whole pipeline.

Backpropped as B9 with the recurrence class spelled out (any
custom-endpoint entity mapping a delegate view is unreadable via list
GET — the diff idiom's hidden assumption), and the operator picked the
fallback remedy: `$filter` list GET stays the primary read, and exactly
the optimization-500 reroutes that record to a new
`client.get_record` key-URL GET. One more probe pinned the missing-record
signature on the key-URL form — a 500 carrying
`NoEntitySatisfiesTheConditionException`, not a 404 and not an empty
list — which the fallback maps to the familiar "missing on tenant"
drift line. Offline tests pin the fallback round-trip, the
fallback-missing path, and that any *other* 500 still raises; live,
the currencies diff went from 500 to a clean exit 0, and the scratch
tenant came down with its recycle.

The baseline expansion order (currencies first) now has its mechanism
end-to-end: the endpoint exposes the entity, the data repo can carry a
`currencies.yaml` (in `baseline/` with the `endpoint:` override, so
accounts sort and seed ahead of it), and diff can verify it.

## Code review over src/: two fixes, two preventive spec rows

A high-effort multi-agent review of `src/` after T29 surfaced four
findings that survived adversarial verification (two more were refuted).
Two were fixed on the spot:

- The template comment in `bootstrap_project.xml` told seed authors the
  ten gain/loss pairs are "required at persist, so the seed data must
  supply them" — the exact opposite of the live-verified `Translation*`
  caveat (friction #37: write-tolerated, read-invisible while
  financial-statement translation is off), which until now lived only in
  the T29 commit message. Anyone following the comment would have built
  seed YAML that diffs dirty forever. The caveat is now in the comment
  itself. Worth remembering the failure shape: a live-verified fact
  recorded in a commit message but not in the artifact it constrains is
  invisible at the moment it matters. Side effect: the comment edit
  changes the package digest, so the next publish against a live tenant
  will republish once.
- `PACKAGE_DESCRIPTION` in `bootstrap.py` was a hand-synced duplicate of
  the XML root `description` attribute — the T29 diff had to edit the
  same sentence in two files, and a divergence would have either desynced
  the tenant-visible description or fired a spurious digest-gate
  republish. The constant now parses the shipped template; the XML is
  the single source.

The other two findings were latent design gaps, not observed failures,
so they went into the spec preventively (operator's call: amend, not
backprop — the §B ledger stays a record of things that actually broke):

- **V20 + T32** — the Bootstrap endpoint's `Currency` entity shares its
  name with the Default endpoint's CM201000 currency-list entity, and
  the per-file `endpoint:` override is optional; a `currencies.yaml`
  that forgets the override would PUT to the wrong screen, report
  success, and bring B8's 422 back behind a clean apply. T32 makes
  `seed.py` hard-error on any entity the shipped Bootstrap template
  serves when the file names no endpoint.
- **V21 + T33** — T29 changed the endpoint's contract shape but kept
  `Bootstrap` at version 1.0.0, so two different contracts share one
  name-plus-version identity; an older acu build touching an
  already-bootstrapped tenant would see a digest mismatch and "repair"
  it by republishing the currency-less contract under the same name — a
  silent downgrade with no version signal in the resulting seed
  failures. T33 bumps to 1.1.0 and lists the ripple targets (templates,
  docstring example, test fixtures, data-repo refs).

## T31: the scratch graduates — and the baseline learns two portability rules

T31 turned the T29 scratch files into the data repo's real `baseline/`:
the 15 SalesDemo subaccounts, the 161-account chart of accounts, and
three financial currencies (CAD, EUR, SGD) through `Bootstrap/1.0.0`,
with the gain/loss pairs pointing at the real accounts (83000 realized,
83100 rounding, 83200 revaluation, 84000 unrealized, all on subaccount
000000) instead of the scratch 8710/8720. The verify bar was the spec's
own goal sentence in miniature: provision a fresh scratch tenant with
zero manual steps and diff clean over everything applied.

The first run died twice, and both deaths became spec rows:

- **B10 — apply order was an alphabetical accident.** The spec named the
  files `subaccounts.yaml`, `accounts.yaml`, `currencies.yaml`, and dir
  expansion sorts alphabetically — so the currencies (whose gain/loss
  pairs reference subaccount 000000) applied before the subaccounts file
  and answered 422. The scratch run had used numbered prefixes precisely
  to force order, and the graduation dropped them. Fixed by adopting the
  prefixes for real (`10-subaccounts`, `20-accounts`, `30-currencies`;
  `uoms` renamed to `90-` as order-independent reference data) and
  recording V22: a record may only reference what already exists at PUT
  time — tenant-native or created by an earlier-sorting file — and
  filename prefixes are the ordering mechanism, because nothing else is.
- **B11 — server-derived fields drift by construction.** The second
  fresh provision applied all 191 records clean and then the diff leg
  lit up with 89 drift lines: `ChartOfAccountsOrder` (the server
  recomputes it from account type; SalesDemo derived 4 where a fresh
  tenant derives 3) and `CashAccount` (a derived flag that only flips
  when a CA cash-account row references the GL account — none exist on
  a fresh tenant). Both are the Translation* shape again: the PUT is
  accepted, the server keeps its own value, and any sourced value diffs
  dirty forever. V22's strip rule now covers server-derived fields, and
  the generator strips both alongside `AccountGroup` (which had already
  422'd — the raw extract carried it while the T24-verified replay file
  had stripped it, a divergence that cost one failed run to rediscover).

Third run: fresh provision exit 0, no drift over six files, and an
independent `acu diff baseline/` confirmed exit 0 — currencies read back
through the B9 key-URL fallback. The scratch tenant came down with its
recycle, and the data repo's baseline now carries the dependency chain
the status board has listed as "open" since day one: subaccounts, chart
of accounts, financial currencies, UOMs.
