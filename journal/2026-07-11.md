# 2026-07-11

## T47: tenant create becomes resumable — the republish route B17 lost

B17 recorded the gap: when T45 retired `acu provision`, it also retired
provision's resumability without re-routing the recovery role T8 had
recorded for it ("re-running provision is the republish route"). The
consequence was live and concrete — tenant Company still ran the
pre-T37 Bootstrap package, the 1.4.0 seed files 404'd against it, and
the data repo's `make diff` had been red since the version bump with no
CLI path to fix an existing tenant short of delete-and-recreate.

T47 restores the route inside `acu tenant create`. Before the ac.exe
create, the command reads the live tenant list (the V4 rule: the gate
verifies state, never a marker). When the login already exists on the
instance it prints `skip create: tenant <login> exists (id <n>)` and
runs the rest of the chain unchanged — recycle, login check, and the
digest-gated bootstrap publish, which is exactly where the resumability
lives: a tenant on stale package content gets a republish, a current
one gets `already published`. A `--id` that does not match the existing
tenant's CompanyID is a hard error naming both numbers (create with an
existing CompanyID is an overwrite in ac.exe, so the guard errors
before anything runs). The fresh path is byte-identical apart from the
one extra list read.

The live verify replayed B17's exact scenario. `acu tenant create --id
2 --login Company` skipped the create, and the T25 digest gate saw the
stale content and republished Bootstrap 1.4.0 — the content-aware skip
doing precisely the job B7 bought it for. The mismatch guard was
checked read-only (`--id 5 --login Company` errored with both IDs and
touched nothing). The first `make diff` after the republish was an
honest red, and a good one: with the endpoint restored, the probe
could finally read, and it reported the single true gap — the T37
open-periods action had never run on this tenant (`action ProcessAll:
not applied`). One `make apply` invoked exactly that action and
skipped everything else; `make diff` exited 0. The data repo is green
again, and re-running `acu tenant create` is once more the documented
recovery route for an existing tenant.

With T47 closed the §T backlog is empty.

## T48: extract machinery — live tenant state to seed YAML, offline

T48 is the first of the three extract-milestone rows the morning spec
amendment queued: the engine and command, fully offline; setup/
synthesis and the live round-trip follow as T49/T50.

`acu extract` is the inverse of apply. A packaged manifest
(`extract_manifest.yaml`, read via `importlib.resources` like the
other package data) drives it: one row per entity carrying the source
endpoint, key fields, destination file, and a strip deny-list or
include allow-list (mutually exclusive; key fields survive both). The
nine M1 rows are the verified GL set — Company, CreditTerms,
Subaccount, Account, Currency, Ledger, GLPreferences, LedgerCompany,
UnitsOfMeasure — and the strip lists carry their reasons as comments:
AccountGroup and RevaluationRateType reference entities outside the
baseline set (B10), ChartOfAccountsOrder and CashAccount are
server-derived (B11), the Translation* currency pairs are
read-invisible while the translation feature is off (T31), and
Ledger.CurrencyID is derived from the company base currency (T34).
The manifest never names a Bootstrap version: rows say `endpoint:
bootstrap` and the loader resolves `seed.BOOTSTRAP_ENDPOINT`, so a
version bump cannot strand a stale ref in package data (V21). The
manifest validator enforces V20 by construction — a row whose entity
the Bootstrap template serves must carry an endpoint, so every
emitted file parses through `load_baseline` without the ambiguity
error.

Emitted files are byte-stable by shaping, not by accident: records
sort by key tuple, fields order key-first then alphabetical, and
None or empty-string values are elided. A test extracts twice — the
second run against a server returning the same records in reversed
order with reversed field order — and asserts byte-identical files.
That property is what T50's cross-tenant leak detector (extract A,
provision B from it, extract B, compare bytes) will stand on.

The B9 read problem applies to extract too, in list form: the plain
list GET on a delegate-view entity 500s. The fallback narrows the
list GET to the key fields via `$select` (delegate fields out of
scope), then reads each record through the key-URL single-record GET.
Whether the `$select` projection actually dodges the optimizer live
is T50's first archaeology question; offline the shape is pinned by a
request-sequence test. The `_OPTIMIZATION_500` marker moved from seed
to client (public, one spelling) with seed importing it back.

Twenty-two offline tests ride a FakeServer that honors `$filter`,
`$select`, and key URLs, so extract and the diff round-trip run
through the real client against canned tables for all nine entities.
The round-trip test is the load-bearing one: extract, parse every
emitted file with `load_baseline`, diff against the same canned state,
zero drift — strip and elision never manufacture drift. The rest pin
skip-if-exists versus `--force`, skip-on-zero-records, dry-run
writing nothing, `--only` filtering by entity name or file stem, the
manifest self-check, and CLI flag wiring. `make check` green at 202
tests.

## T51: one runner for the pre-commit audit

The T48 build run surfaced a mechanization candidate: every build's
pre-commit audit re-ran three hand-copied commands — `check-ascii
src/`, `check-extras.sh`, and the §V.21 version-parity one-liner
pasted from `.spec/check-extras.md`. T51 folds them into
`.spec/scripts/check-all`: no arguments runs every recipe, a leg
argument (`ascii`, `extras`, `parity`) runs one, each failing leg
streams its surviving output followed by a `check-all: <leg> FAIL`
line, and the exit code aggregates. The parity sweep now lives only in
the script — the doc's §V.21 recipe command flips to `check-all
parity`, so the one-liner has a single spelling while per-recipe
commands and bail messages stay for /sdd:check granularity.

The mechanization immediately paid for itself: the real-tree test
failed on the first run because the hand-run one-liner had a latent
scope gap — `grep -r` descends into `__pycache__`, and compiled `.pyc`
files echoing Bootstrap refs survive the literal filter as "Binary
file matches" lines. The script adds `-I` and
`--exclude-dir=__pycache__` (compiled artifacts echo source refs and
are not sources), the same move that made check-ascii's exemptions
code instead of eye-applied filters. A second self-inflicted trap:
the test that plants a stale ref had put the literal inside `tests/`,
which is in the sweep's own scope — the test now builds the stale
string by concatenation so the scanned tree never carries it. Eight
offline tests pin per-leg violations on a synthetic repo tree, the
real tree passing, single-leg selection, and unknown-leg rejection.

## T49: extract synthesizes setup/ and the feature closure

T48 extracted what a keyed GET can see; T49 covers the two file kinds
that have no records to dump. An action file describes a verb, not
rows — the action already ran on the source tenant, and all it left
behind is the state its `done_when` probe reads. So extraction runs
backwards through that probe surface: three `kind:`-dispatched
synthesizers (the manifest's `setup:` rows name kind and destination,
and the `SetupSynth` model rejects a kind with no registered
synthesizer) each read the live state the action created and derive
the action file that would recreate it.

`financial-year` reads the `FinancialYearSettings` singleton and emits
the `GeneratePeriods` file, keeping only the date part of the live
`BegFinYear` (a DateTimeValue comes back as a full ISO datetime; the
action record wants the quoted date). `master-calendar` reads the
`MasterCalendar` year range and emits `GenerateCalendar` with
`FromYear`/`ToYear` at the range ends, probing the *last* year's
`CompanyCalendar` — generation completing through the range is the
stronger done evidence. `open-periods` reads `CompanyPeriod` filtered
`Status eq 'Open'` and sets `OrganizationID` from the extracted
Company's `AcctCD`, so the reference resolves inside the emitted set
(V22's in-set closure rule). A tenant with generated but unopened
periods gets the file skipped plus a stderr warning — that half-state
replays into a tenant that cannot post GL (the B13/B16 shape), and a
silent skip would hide it.

The feature closure works the same way but for a reason worth
recording: a live `FeaturesSet` read over the contract API is
verified-impossible (keyless BqlDelegate view — the T3-era finding
still holds), so `bootstrap/features.yaml` cannot be extracted, only
derived. `render_features` emits the built-in six (imported from
`bootstrap.DEFAULT_FEATURES` — one spelling) plus the union of the
manifest's `features:` gates over entities whose destination file is
in the output set — written this run or already on disk, so a re-run
that skips existing files still closes over them, and an entity with
zero live records drops its gate. True read-back (a GI-backed
Bootstrap entity, OData against the DAC, or sqlcmd with a V1
amendment) stays deferred.

Ruff pushed one refactor: the three passes over one set of knobs
(out_dir, only, force, dry-run) tripped both the complexity and
argument-count limits as functions, so they became methods on an
`_Extraction` run-context object — the knobs are constructor state,
the passes read them. Eighteen new offline tests bring the suite to
220: the synthesized docs asserted whole against the FakeServer state,
the round-trip extended so every action file parses to `ActionFile`
and diffs clean through its probe, byte-stability now sweeping all
thirteen emitted files, the features union and its drop/keep edges,
and the none-open warn path. T50's live round-trip is the remaining
extract row.

## T50: the live round-trip — extract proves itself the inverse of apply

The last extract row ran against the real instance, and the shape the
task predicted ("expect strip-list discoveries — budget archaeology
loop") is exactly what happened: one loop, three leaks, all one class.

Before any test code, two archaeology probes ran read-only against the
existing Company tenant (V12). First, B9 is alive on this build: the
plain list GET on Bootstrap Currency 500s with the optimization marker,
and the $select-narrowed list GET answers 200 — so extract's fallback
is the load-bearing read for Currency, not a belt-and-suspenders path.
Second, and the surprise of the day: that $select probe came back with
172 rows. The Bootstrap Currency entity serves the entire tenant-native
ISO currency list, with an IsFinancial field marking the configured
rows (true on the four real ones, false on the ~168 bare list entries).
Extract dumps what the tenant serves, so the currencies file carries
all 172 — replayed as no-op upserts and re-extracted byte-identically,
which the round-trip then proved, but it is noise as config-as-code and
it is where extract's wall-clock goes (three of the four legs walk the
per-key fallback, 172 sequential GETs each). A completeness probe also
confirmed unpaged list GETs match a $top=10000 sweep at every scale
observed (12 to 172 rows). Both findings are pinned in the test file's
docstring.

The test itself (tests/e2e/test_extract_roundtrip.py, eight sequential
legs, two scratch tenants bracketed by a session fixture that always
deletes and recycles per V5): tenant A created and configured from the
data repo, extract into a/; tenant B created fresh and configured from
a/ alone; diff over a/ reads B clean; extract from B into b/ must be
byte-identical to a/; and a hand-built GL batch (the T37 recipe — wrap()
is scalar-only, Details is a list, ReleaseJournalTransaction, poll by
key URL) releases to Posted on B, proving the replayed setup chain
whole end to end (the B16 class made a permanent regression net).

First run: six of eight green, and the byte-compare plus the diff leg
failed on the same root — the wholesale a/-vs-b/ diff (the surviving
pytest tmp dirs are better archaeology than the first failing file)
gave the complete leak inventory in one pass. SubaccountID and
AccountID are server-assigned surrogate identities off an
instance-global sequence: A extracted 699-713, B re-extracted the same
fifteen subaccounts as 714-728 — another tenant's row numbers, the
purest B11 specimen yet. CreatedDateTime leaked on Account (the insert
stamp, sibling of the already-stripped LastModifiedDateTime) and
LastModifiedDateTime on Ledger (whose strip list only carried
CurrencyID). Three manifest strip rows with load-bearing reasons closed
both legs; the re-run went eight for eight in 13:26, and the full
make e2e — round-trip plus lifecycle, fourteen tests — finished green
in 22:27.

One environment note for the record: the sibling data repo is now
acumatica-salesdemo (161-account SalesDemo chart, LAB5 org), replacing
acumatica-baseline; the gitignored symlinks were relinked and the whole
live tier verified against the new set. SPEC C and CLAUDE.md still name
the old repo — a spec amendment, not a build's write. And the timing
question the 172-row discovery raises has a better answer than a
smaller demo company: tenant creation dominates the fixed cost and the
currency list is tenant-native, so shrinking the chart of accounts buys
little — a record-level manifest filter (IsFinancial eq true) is the
lever that shrinks both the noise and the wall clock.

## T52: the manifest record filter — currencies become config-as-code

The lever T50 identified, built. `EntitySpec` gains an optional
`filter:` — an OData `$filter` fragment that rides both list reads, the
plain GET and the B9 `$select` fallback, so the two paths always serve
the same record set and the per-key walk only visits filtered keys.
The Currency row sets `IsFinancial eq true` with the T50 archaeology
recorded as the comment: Bootstrap Currency serves the whole
tenant-native ISO list (~172 rows), and IsFinancial is the flag that
separates the four configured currencies from the ~168 bare list
entries.

Offline, the FakeServer's `$filter` handling grew a second literal
form — quoted values compare verbatim as before, bare `true`/`false`
compare case-folded against the JSON booleans — and the Currency table
gained a non-financial JPY row so the narrowing is observable. The B9
request-sequence test now pins the filter on both list reads and JPY
never getting a key-URL walk; a new `_fetch` test pins the plain-GET
path on a non-delegate entity; the manifest self-check asserts exactly
one filter row exists and names it. Strip and include semantics are
untouched — the same test still shows Translation* pairs stripped off
records the filter let through.

Live, the payoff was exactly as budgeted: `make e2e` green fourteen for
fourteen in 21:16 (down from 22:27), both extracts emitting
`30-currencies.yaml (4 records)` instead of 172, the a/ versus b/
byte-compare clean, and the GL batch still posting on the replayed
tenant. The extracted currencies file now reads as four intentional
records with real gain/loss accounts — configuration, not an ISO
almanac.

## B18 + T53/T54: the backlog sweep that caught its own spec lying

A `/sdd:build --all` sweep over the last two rows. T53's plan step
probed the row's premise before touching anything and found it stale
at birth: the row (authored 14:07) claimed `.env.gpg` sat at the repo
root "untracked + unignored", but commit 8ad8441 had deliberately
tracked it at 10:06 the same morning. Executed as written, the
gitignore-only edit would have passed the row's literal verify —
`git check-ignore` answers from patterns, not the index — with the
ciphertext still riding every future tip. Ignored-but-tracked is the
worst of both: green gate, standing concern. Classified as an unspec
edge and backprop'd: B18 recorded, and V17 gained a premise sibling —
spec text asserting repo or live state must be probed at authoring
(`git ls-files` class), because a stale premise plans the wrong edit
while the literal verify gate greens. The amended T53 added the
untrack leg and an ls-files-empty verify.

Executing it hit a second trap worth keeping: the build's path-scoped
commit idiom (`git commit -- <paths>`) silently *reverted* the staged
`git rm --cached` — commit-with-pathspec takes the working-tree state
of the named paths, and the file intentionally still exists on disk,
so the first commit claimed an untrack it had quietly undone. The
post-commit re-probe caught it (`git ls-files` still listed the file);
restaging the deletion and amending — a plain amend commits the index
— landed it for real. Lesson: for index-only deletions, pathspec
commits are the wrong instrument, and the verify probes belong after
the commit, not only before it.

T54 is the mechanization T50's build queued: `make e2e
FILE=<path-or-stem>` scopes the live tier to one file.
`$(wildcard)`/`$(firstword)` resolve FILE as a path, then
`tests/e2e/<FILE>`, then `tests/e2e/<FILE>.py`; no match errors out
before pytest naming the tried forms; bare `make e2e` is
byte-identical (`-o addopts= -m e2e` both modes, the tier stays
opt-in). Verified without going live: `make -n` shows the resolved
target for stem, path, and bare invocations; the bogus leg dies
pre-pytest; `--collect-only` proves 8 tests for the single file
against 14 for the tier. The recipe rewrite also fixed the stale
decrypt hint — it pointed at `make decrypt` in
`../acumatica-baseline`, wrong on both halves since the rename and
the `.env` real-file migration.

The §T backlog is empty again.

## T55 + T56: shell completion and the --type choice — the backlog closes again

Two small surface tasks, both click-native, both about the CLI telling
the truth at the prompt instead of at the server.

T55 adds the `--completion` global: `acu --completion fish` (or bash,
or zsh) prints click's completion script on stdout and exits eagerly
like `--version`; a bare `--completion` resolves the dialect from the
`$SHELL` basename, and anything unresolvable errors naming the
supported set. Enabling stays the user's move — source the output.
The interesting constraint is V23: completion fires per keystroke, so
the whole path must be local-only. The dynamic piece honors that by
construction — `extract --only` completes entity names off the
*packaged* manifest (package data, no live read), and `apply`/`diff`
path arguments complete as plain files because click's `Path` type
already does that natively; nothing new to write, only a test to pin
it. The test file wires `load_instance`, `AcumaticaClient`, and
`TenantManager.__init__` to assert-fail, then drives click's real
`ShellComplete.get_completions` machinery — any live-path touch fails
the suite loud. One environmental footnote: click's bash emitter warns
on macOS's system bash 3.2 (stderr, harmless); the script still emits.

T56 turns `tenant create --type` from a free string into
`click.Choice(["SalesDemo", "T100", "U100"])` — the dataset folders
verified on the box (docs/ac-exe.md), with `System` deliberately
excluded as the system-tenant dataset. A non-member now dies at parse
time naming the allowed set, before any SSH; previously a typo rode
ac.exe all the way to a half-created tenant. Omitted still means the
clean tenant — the option defaults to None and the call site maps it
to ac.exe's empty CompanyType, so `mgr.create`'s shape is unchanged
(V7). The help metavar carries the exact names.

Both landed offline-verified (`make check`, 235 tests) with
`.spec/scripts/check-all` clean before each commit. The §T backlog is
empty again.
