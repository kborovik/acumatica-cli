# 2026-07-13

## T61 build: Bootstrap/1.5.0 distribution setup entities, live-iterated on LAB5

T61 (gh issue #10) grew the Bootstrap endpoint from ten entities to
eighteen so the LAB5 demo manufacturer (data repo `acumatica-devops`,
Lab5 Gateway Works) can carry distribution config. The probe evidence in
the issue named five module-prefs screens (IN/AP/AR/SO/PO) gating every
distribution entity, plus AvailabilityCalculationRule (ItemClass
hard-requires one, an empty tenant has zero), PostingClass, and
CashAccount (Default 404). The build was the by-now-standard loop —
aspx reads off the box for graph/PrimaryView bindings, minimal field
sets, then live PUT iterations reading the 422/500s — and the live
tenant corrected the plan four times:

- **INSetup is not defaults-only.** The GLPreferences precedent (two
  no-default fields) suggested prefs singletons just need to exist. IN
  refused: the insert named six no-default fields — the
  INProgressAcctID/INProgressSubID WIP pair and four reason codes
  (Issues/Receipt/Adjustment/PI). That pulled **ReasonCode (CS211000)**
  into the endpoint as an eleventh entity — no Default-endpoint entity
  exists — and a WIP account (12300) plus five reason-code rows into
  the data repo. Bonus finding off the Usage 422: no 'Physical
  Inventory' usage on this feature set (the PI selector takes
  Adjustment-usage codes), and 'Assembly/Disassembly' exists — the kit
  scenario's INASSEMBLY code seeded ahead of need.
- **POSetup names one required field** (RCReturnReasonCodeID, a Vendor
  Return usage code); AP/AR/CA setups saved with an echo flag alone;
  SOSetup accepted DefaultOrderType SO — so order type SO **does**
  pre-exist on an empty tenant once SOSetup first saves (the plan's R1
  risk closed cheap).
- **CashAccount gates on CASetup** (CA101000) — the issue had it as a
  conditional follow-up; it is not conditional. CAPreferences became the
  twelfth entity; TransitAcctId (DAC spelling off the aspx, lowercase d)
  + TransitSubID are its no-default pair; data adds a 10200 Cash In
  Transit account.
- **Warehouses are implicit when MultipleWarehouses is off.** Saving
  INSetup auto-creates a warehouse named MAIN reserved for transit, and
  any user warehouse insert — MAIN or a fresh WH01 — 500s "the
  warehouse cannot be selected; it is used for transit". The data
  repo's 55-warehouse.yaml was deleted, not fixed: there is no
  user-creatable warehouse on this feature set, documents will ride
  the implicit one (plan risk R2 closed).

**B23 fell out of the first clean diff.** INPreferences is keyed on
HoldEntry (boolean — the only echo field), and `_filter_for`
string-quotes every key literal: OData answered 500 "binary operator
with incompatible types 'Edm.Boolean' and 'Edm.String'". Latent since
the beginning — every prior baseline key was a string. Fixed in
`_filter_for` (bool/number bare, string quoted, so numeric-looking
codes keep their zeros), pinned offline, backpropped as §B.23 with the
rule folded into V4.

**A publish-cwd hazard, recorded for a future invariant.** One
republish ran from this repo's root instead of the data repo. Discovery
found this repo's own `.env`, `load_features` saw no
`bootstrap/features.yaml` (the data symlinks are gone since the T53-era
cleanup), silently fell back to DEFAULT_FEATURES — and the publish
**downgraded the live tenant's feature set** (SubAccount off → the
whole baseline 403s at GL203000). Recovered by republishing from the
data repo. The class: `tenant create` against an existing tenant +
absent features.yaml = silent feature regression behind a green
publish. Worth a guard (warn or refuse when the file is absent and the
tenant already exists) — left as a finding, not yet a §T row.

Two smaller findings: ItemClass PurchaseUOM/SalesUOM equal to BaseUOM
are not returned by the Default endpoint (B11 phantom class — omitted
from seed YAML), and ItemType allows only 'Finished Good' with
Manufacturing off. The KitAssemblies feature flag (data-repo
features.yaml addition, package republish) cleared the 403 on
KitSpecification/KitAssembly — flag name verified live.

End state: `make check` green (250 offline), full LAB5 apply green
across 24 files, `acu diff` zero drift, and every distribution +
transaction entity (Vendor, Customer, StockItem, KitSpecification,
KitAssembly, SalesOrder, PurchaseOrder, PurchaseReceipt, Bill,
Shipment, SalesInvoice, Payment) answers 200 — the surface the T62
scenario runner needs is fully unlocked.
