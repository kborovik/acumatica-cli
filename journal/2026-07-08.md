## Morning — recycle unblocks the tenant; the "foot-gun" was a stale-map artifact; first-login password is the real bootstrap wall

Ran the app-pool recycle that was pending operator approval yesterday
(`Restart-WebAppPool -Name AcumaticaERP` over SSH). It cleanly resolved the
visibility problem **and corrected two conclusions from last night** — both
worth recording, because "the thing I was sure of was an artifact" is the
honest shape of this kind of work.

**The recycle works — tenant 3 is now real.** After the recycle the sign-in
page lists `Scratch`, and REST tenant routing turns *strict*. So the fix is
verified: `acu tenant create` must recycle the `AcumaticaERP` app pool after
`ac.exe` returns, or the new tenant stays invisible to the running app until
something else recycles it.

**Correction to Surprise #2 — the silent foot-gun was the stale map, not the
API.** Last night I "proved" that REST login accepts *any* tenant name and
silently lands on the default. Re-tested post-recycle, that's no longer true,
and the difference is the tenant map being current:

- `tenant='DefinitelyBogus'` (unknown, non-empty) → **500
  `PXUndefinedCompanyException`** ("A proper company ID cannot be determined").
  No silent fallback anymore — unknown names are rejected.
- `tenant='Scratch'` (valid name) → routes to tenant 3 and enforces *tenant
  3's* credentials (returned invalid-credentials, **not** a silent success on
  tenant 2's password). Proof the routing is per-tenant now.
- `tenant` omitted or `''` → still 204 on the default tenant. This is the only
  residual silent behavior, and it's expected Acumatica semantics (no tenant =
  default tenant).

So the real, narrower rule for the CLI: **always send an explicit, valid
tenant name — never omit it, never trust an empty value.** The dramatic
"writes to a nonsense tenant" scenario was specifically a symptom of skipping
the post-create recycle (tenant in the DB, absent from the loaded map). With a
current map, a wrong name errors loudly. A post-login tenant guard is still
worth having as defense-in-depth, but it's a seatbelt, not a firewall.

**The real bootstrap wall — first-login password.** With tenant 3 finally
reachable, the honest virgin-tenant login story emerges (last night's "login
is NOT a blocker" was itself an artifact — those probes hit tenant 2, which is
long past first login):

- A freshly created clean tenant seeds `admin` with password **`setup`** and
  `PasswordChangeOnNextLogin = 1` (confirmed in `dbo.Users`).
- The contract REST login **refuses to perform the change**: passing
  `newPassword` alongside `setup` still returns *"You need to change your
  password. Sign in by using your browser to set a new password."* The
  `/entity/auth` endpoint can authenticate but not clear the first-login flag.
- Repeated wrong-password attempts **lock the account** (`LockedOutDate` gets
  stamped) — my probe loop tripped it on tenant 3.

Consequence: **the first-login password change is a bootstrap step the
contract REST API cannot do.** `acu provision` has to handle it out of band —
either drive the screen-based login (`Login.aspx`, which is what the error
tells you to use), or, as part of `acu tenant create`, clear
`PasswordChangeOnNextLogin` and set a known admin password in the DB so the
tenant comes up automation-ready. Design call deferred to the provisioning
work; documented in `docs/rest-api.md`.

**Next.** (1) Decide the first-login password strategy (screen login vs. SQL
reset at create time) and bake it into `acu tenant create`; unlock tenant 3.
(2) With a genuinely loginable virgin tenant, re-run the entity/CustomizationApi
probe against tenant 3 to confirm last night's map (all gathered against
tenant 2) holds on a real fresh tenant. (3) Author the bootstrap package. The

## Afternoon — screen-login password change prototyped; virgin-tenant map confirmed

The first-login wall from this morning fell to ~60 lines of httpx + stdlib
`HTMLParser` — no browser, no Selenium, no SQL poking. The `Login.aspx`
WebForms flow is fully scriptable:

1. `GET /Frames/Login.aspx` → harvest `__VIEWSTATE` & friends + session
   cookie.
2. `POST` back with `cmbCompany=Scratch`, `txtUser=admin`, `txtPass=setup`,

   `txtNewPassword` / `txtConfirmPassword` now present** — that's the
   password-change view.
3. `POST` again with old + new + confirm filled → redirect to `/Main`,
   signed in. REST login with the new password: **204**.
   `PasswordChangeOnNextLogin` flipped to 0 in `dbo.Users`.

**Errors & traps (all hit):**

- First attempt posted the wrong button — the page has *four* submit inputs
  and `mfLoginButton` (multi-factor) matches "login" first; the right one is
  `ctl00$phUser$btnLogin`.

  silent re-render, no change. Both must be posted.
- The page markup *always* contains alarming hidden divs ("Last update was
  unsuccessful", "customization failed to apply automatically after the
  upgrade") — static templates, not responses to your POST. Cost me a
  double-take.
- The login JS sets an `isReal` click flag, but it's client-side theater — a
  plain postback is accepted.

**Decision:** screen-flow automation wins over the SQL-reset alternative — it
uses a supported surface, exercises the same path a human would, and leaves
`dbo.Users` untouched. `acu tenant create` should chain: `ac.exe` → app-pool

## Evening — `acu tenant create` chains the whole thing; two more ac.exe landmines defused

Wired `ac.exe → recycle → login-ready` into `acu tenant create` and made it
work end to end against a real fresh tenant (id 4, `Scratch2`): create,
app-pool recycle, REST login verified — one command, no manual steps.
`acu tenant delete` round-trips it too. But getting there tripped two more
CompanyConfig landmines that the ~40-second happy path from yesterday hid.

**Landmine 1 — `-iname` isn't enough; you also need `-h` (instance path).**
The first CLI create blew up mid-run:
`System.ArgumentNullException: … path1` in
`AcumaticaConfig.Installation.WebCompanyConfig()`, and the whole thing rolled
back. Yesterday's manual creates happened to dodge it (they ran far enough to
insert data before the web.config stage, or the earlier invocation differed);
the moment it ran clean from the CLI it surfaced. Fix: pass
`-h:"C:\Acumatica\AcumaticaERP"` (found via the IIS site's physical path in
the registry). Added `instance_path` to the instance YAML alongside
`instance_name`.

**Landmine 2 — delete is a minefield of its own.** Two nested traps:

- The sub-key is **`Deleted`**, not the officially documented `Delete`.
  `Delete=Yes` is silently ignored — and the run then treats the tenant as a
  *data-insert target and overwrites it*. (This is what mangled tenant 4 into
  a half-System husk mid-debug.)
- Even with `Deleted=Yes`, a bare `CompanyID=N;Deleted=Yes;` aborts at the
  preflight (`dataCpnTest` → `CompanyMaint.CheckAndUpdate`) with
  **"The system company cannot be removed."** The preflight reads the missing
  `CompanyType` as the system tenant. Passing the full spec
  (`ParentID=1;CompanyType=Custom;Deleted=Yes`) clears it. `delete()` now
  looks the tenant up first and echoes back its real type.

**Correction — `LoginName` *did* take, I looked in the wrong column.**
Yesterday: "LoginName=Scratch didn't become CompanyCD, got auto-named
Company2." Today: `LoginName` lands in `dbo.Company.CompanyKey`, which is
exactly the sign-in name and the REST `tenant` value. `CompanyCD` is just an
auto-generated internal code. So the tenant was named correctly all along;
`acu tenant list` now reads CompanyKey as the login name.

**Simplification — skip the screen dance entirely on create.** The
`-aun`/`-aup`/`-auc` admin-preset flags set the new tenant's admin password
up front with no must-change flag, so REST login works straight after the
recycle. The `Login.aspx` form flow from this afternoon still ships as the
fallback (for tenants born without the preset), but the create path no longer
needs it — `initialize_admin_password()` just confirms the REST login and
returns "already initialized". Nice outcome: the fussy WebForms scraping is
now belt-and-suspenders, not the critical path.

Net: `acu tenant create --id N --login NAME` is a real one-liner now.

## Late — tenant login names can be domain-style (`lab5.ca-dev1`)

Question that matters for the naming scheme: can a tenant's sign-in name be a
domain-shaped string like `lab5.ca` / `lab5.ca-dev1`? Tested the hardest case
(dot **and** hyphen) end to end on the box: `acu tenant create --id 4 --login
lab5.ca-dev1` → ac.exe accepted `LoginName=lab5.ca-dev1` (exit 0), it landed
**verbatim** in `dbo.Company.CompanyKey`, `acu tenant list` shows it as the
sign-in name, and a REST login with `{"tenant":"lab5.ca-dev1"}` returned **204**
under strict post-recycle routing (resolved the dotted name, not a silent
fallback). Clean delete round-trip after.

Bounds worth stating: safe alphabet is alphanumerics + `.` + `-`. The two
characters to avoid are `;` and `=` — the `-company:"…;LoginName=X;"` string
uses them as delimiters, so a name containing either would corrupt the ac.exe
invocation (a validation/escaping concern for `create` if names ever come from
untrusted input). Spaces / `@` / `/` untested. `CompanyKey` carries the pretty
name; `CompanyCD` stays the auto code (`Company3`). So per-environment tenant
names — `lab5.ca-dev1`, `lab5.ca-prod`, … — are viable.

## CLI output standards — rich, two audiences, one code path

Housekeeping before the CLI grows more commands: defined output standards
(`docs/cli.md`) and adopted [rich](https://rich.readthedocs.io), replacing the
bare `print()`s scattered across `cli.py`/`seed.py`.

The framing that made the design easy: the CLI has **two audiences** — humans
at a terminal, and LLM agents / scripts reading piped output. rich's `Console`
serves both from one code path: TTY → colors, rounded tables, spinners;
piped → plain deterministic text (no ANSI, no box-drawing). Verified both
renderings live against acu-dev1.

Mechanics: new `output.py` is the only module that creates a `Console`
(stdout = data/results, stderr = status/progress/errors; `✓`/`!`/`✗`
prefixes survive piping so agents can grep them). Long ops (`tenant create`
steps) wrap in `output.step()` — spinner on TTY, plain stderr line piped.
A top-level handler in `cli.main()` turns `RuntimeError`/`httpx.HTTPError`
into a one-line `✗ message` + exit 1 (`ACU_DEBUG=1` re-raises the traceback)
— previously every SSH/REST failure dumped a full Python traceback. The
`diff` exit-1-on-drift contract is unchanged (re-verified: `✓ no drift`,
exit 0). Enforcement: ruff `T20` now fails the build on any bare `print()`.

Two details worth recording: markup/emoji/highlighting are disabled on both
consoles — apply lines like `PUT Currency [CAD]` contain brackets rich would
otherwise eat as markup; and `tenant list` switched from tab-separated rows
to a proper table (nothing parsed the tabs — checked Makefile/scripts first).

Follow-up polish: login now goes through `_checked()` like every other call,
so a failed login shows Acumatica's own message (`✗ POST …/auth/login -> 500:
A proper company ID cannot be determined for the request.`) instead of
httpx's generic status line with an MDN link.

## pydantic as the model standard; instances/ folded into acu.toml

Two related structural moves, no live-instance behavior change (50 offline
tests green; `make check` clean):

**pydantic replaces dataclasses everywhere.** New `models.py` defines the
standard: every structured value crossing a boundary inherits `Model`
(`BaseModel` with `frozen=True, extra="forbid"`). `Instance`, `Tenant`, and
`BaselineFile` converted. What this buys over the frozen dataclasses:

- *Validation moves into the model.* `BaselineFile`'s hand-rolled checks
  (required fields, key-listification, every-record-carries-its-keys) became
  a `field_validator` + `model_validator`; `Instance` normalizes
  `base_url`/`endpoint` slashes in validators instead of in `load_instance`.
- *Typos are now errors.* `extra="forbid"` rejects unknown fields in both
  `acu.toml` tables and baseline YAML — previously a misspelled `db_nmae`
  or stray baseline key was silently ignored.
- `dataclasses.replace` → `model_copy(update=...)` (CLI `-t` override).
- `validation_summary()` renders pydantic's `ValidationError` into the
  one-line `✗`-friendly form docs/cli.md wants.

One convention shift: pydantic models take keyword args only, so `Tenant(1,
"System", ...)` in tests became explicit kwargs — noisier but self-labeling.

**`instances/*.yaml` → `acu.toml`.** One TOML config file at the repo root
instead of a YAML file per instance: `[instances.<name>]` tables plus a
top-level `default_instance`. The CLI's `-i` default moved out of the code
(`cli.py` no longer hardcodes `acu-dev1`) and into config where it belongs.
Reading is stdlib `tomllib` — PyYAML stays only for baseline files. TOML
literal strings (`'C:\Program Files\...'`) also read better for the Windows
paths than YAML's escaping.

## Repo split: acu is the tool; acumatica-baseline is the first data repo

Objectives redefined. This repo is now **only the `acu` CLI**; the data it
applies moved to a new self-contained sibling repo, `acumatica-baseline`:
`baseline/*.yaml` (what), `acu.toml` (where), `.env.gpg` (secrets). Clone the
data repo, decrypt, run one command — the "byte-identical tenant" definition
of done now belongs to data repos, of which there can be many.

The discussion that led here started as "should `acu config get/set` manage a
config file in `~/.acu/`?" — rejected for this repo's original shape (in-repo
`acu.toml` *is* the config-as-code), but the underlying itch was real: the
tool and the data wanted to be separate. Splitting the repos scratches it
without giving up version-controlled config: the data repo carries everything,
and `~/.acu/` stays unnecessary.

Mechanics — the seam was already clean (`apply`/`diff` take explicit file
paths; `baseline/*.yaml` was hardcoded only in the Makefile), so the only
code change is discovery in `config.py`: `repo_root()` (walk up from cwd to
`pyproject.toml`) became `data_root()` (walk up to the first dir containing
`acu.toml`, hard error with a pointer to data repos if none — no silent cwd
fallback). `.env` loads from the same directory as the found `acu.toml`.
No `ACU_ROOT` env override for now — YAGNI with one data repo.

How the data repo invokes the tool: `uv tool install --editable
~/github/acumatica-devops` (its `make install`), putting `acu` on PATH with
tool-repo source edits live. Rejected: `uv run --project ../acumatica-devops`
(hardcodes a relative sibling path into every Makefile target) and `uvx
--from` (re-resolves per run, not editable).

What moved where: `baseline/`, `acu.toml`, `.env.gpg`/`.env`, and the
`tenants/apply/diff/encrypt/decrypt` Makefile targets went to
`acumatica-baseline` (plain `mv` — history stays here). This repo keeps the
Python project, `docs/`, `journal/`, `tests/` (already fixture-based, only
the `pyproject.toml` sentinel in `test_config.py` needed changing; two new
tests cover parent-dir discovery and the missing-`acu.toml` error), and
`scripts/dump-swagger.sh` + `schemas/` (env-var driven dev aid).

Post-split dev ergonomics: the tool repo carries gitignored symlinks —
`baseline`, `acu.toml`, and `.env` → `../acumatica-baseline/` — so `acu`
still runs from here during development. `.env` had to be linked alongside
the two requested paths because `data_root()` loads it from the directory
where it finds `acu.toml`; without it, live commands from the tool repo fail
with "ACU_PASSWORD not set". Ignored rather than committed: a clone without
the sibling repo would just get dangling links.

Small CLI fix: `acu diff baseline/` crashed with a raw `IsADirectoryError`
traceback — click's `Path(exists=True)` accepts directories, and
`load_baseline` then `open()`s one. `apply`/`diff` now expand directory
arguments into their sorted `*.yaml` files (`expand_files` in cli.py; empty
dir is a clean error), so `acu diff baseline/` matches the Makefile's
`baseline/*.yaml` glob behavior without depending on the shell.

**Landed (offline-verified, committed per task):**

- **T2 — bootstrap machinery.** `bootstrap.py` builds the package zip from
  in-repo `bootstrap_project.xml` and drives
  import → publishBegin → publishEnd-poll via new CustomizationApi methods
  on `AcumaticaClient` (same cookie session). `acu bootstrap` publishes to

  poll tolerates transport errors (publish restarts the app domain).
- **T1 — `acu provision`.** One command: tenant create (skipped if the

**T3 — the verification that answered "no."** Question from the spec: does
CS100000 (Enable/Disable Features) accept writes through a custom endpoint?
Getting the answer took reverse-engineering how custom endpoints actually
work, because the first `acu bootstrap` against a scratch tenant failed
with a misleading "previously published project cannot be found" — which
turned out to be the CustomizationApi's way of saying the import had
already failed. Findings, all live-verified (see docs/rest-api.md):

- **The CustomizationApi reports failures in-band**: every call answers
  200; errors exist only as `log` entries with `logType: "error"`. Our
  `_checked` (HTTP-status-based) was blind to it — `AcumaticaClient` now
  has `_checked_log` and the import raises on in-band errors.
- **project.xml's root is `<Customization level description
  product-version>`**, not `<Project>` — verified against the training
  package Acumatica ships on the box itself
  (`HelpAndTraining/T270/PhoneRepairShop.zip`, same build). The project
  name comes from the import call. None of the shipped packages contains
  an endpoint item, so that serialization stays unverified —
  `PX.Api.ContractBased.Common.dll` carries `*.endpoint` file globs, which
  is the lead to chase.
- **A custom endpoint is just tenant-scoped rows in five tables**
  (`EntityEndpoint`, `EntityDescription`, `EntityFieldDescription`,
  `EntityMapping`, `EntityActionDescription`). Installed the Bootstrap
  endpoint as raw SQL rows on the scratch tenant and it went live in
  `GET /entity` instantly. The constants came from PowerShell reflection
  against the server's own DLLs: `ObjectType` T/L/D/R, field types are the
  wrapper names (`BooleanValue`, `StringValue`, …), mapping keys are
  `E/<entityId>/<fieldId>` → (view, DAC field), `SystemContractVersion`
  must be 4 on this build.
- **Landmines:** built-in endpoints materialize into the same tables under
  CompanyID 1 (EntityIds allocated globally — a colliding id kills the
  tenant's whole contract API with "An item with the same key has already
  been added"); endpoint metadata is cached per app domain (recycle to
  refresh); a malformed row 302s every `/entity` request on that tenant.
  Debugging visibility: flip `EnableFirstChanceExceptionsLogging` in
  web.config and read `App_Data\firstchanceexceptions.log` — the API's
  302-to-`/ui/error` carries nothing.
- **The verdict:** with the endpoint fully correct (swagger 200, field
  echoed in responses, tenant-scoping enforced), `PUT Features` returns
  200 and persists nothing — `FeaturesSet` stays empty. The primary view
  is a keyless BqlDelegate view; the graph save is a silent no-op. GET
  dies in `CannotOptimizeException` (BqlDelegate — CompaniesStructure's
  disease). Exposing and invoking the graph's `Insert` action:
  `PXInvalidOperationException`. **Features cannot be enabled through the
  contract API, full stop.** The spec's fallback — a C#
  `CustomizationPlugin` flipping `FeaturesSet` on publish — is now the
  route.
