# 2026-07-12

## T59 live probe: GL201100 org-scoped reads have no accessible org-context handle over the contract API

T59 (gh issue #8) asked to org-parameterize the CompanyPeriod /
CompanyCalendar read surface so extract emits `setup/30-open-periods.yaml`
and diff's calendar/period `done_when` probes fire on a multi-org tenant —
B22's class, where the org-scoped GL201100 views answer `200 []` because
the contract session carries no org context. The task row offered two
approaches ("org-filtered read or per-org iteration") and flagged a
contract shape change (Bootstrap version bump per V21) if needed. Per the
build gate that a multi-org verify is the only real proof (V26), the work
opened with a live probe against a genuinely multi-org tenant before any
code. The probe is what this entry is about: every lightweight mechanism
was ruled out, and the premise that a readable org-parameterized surface
exists turned out to be unsupported.

**The target.** The `.env` `SalesDemo` tenant (id 3, type Demo) is
unconfigured — `Company` returns 0, the GL forms 500
`PXSetupNotEnteredException`. The real multi-org tenant is `SalesDemo2`
(id 4, type SalesDemo): `Company` lists three organizations (CAPITAL,
PRODUCTS, SERVICES), `LedgerCompany` 14 org-ledger links, `MasterCalendar`
sixteen years (2012–2027), `FinancialYearSettings` a singleton (BegFinYear
2012, 13 periods, Month). A fully configured multi-org tenant — the
calendar exists.

**B22 reproduced, then localized.** In a plain session (login with tenant
only), `CompanyCalendar`, `CompanyPeriod` (Bootstrap) and
`CompanyFinancialPeriod` (Default) all answer `200 []`, while the
non-org-scoped `MasterCalendar` returns its sixteen years and
`FinancialYearSettings` its singleton. So the empty answer is org-context
blindness, not absent data — the per-org calendars/periods are there,
invisible to a context-less session.

Mechanisms tried and ruled out, each read-only:

- **Grid `$filter` on the org field.** `CompanyCalendar?$filter=
  OrganizationID eq 'CAPITAL'` → `[]`, same for PRODUCTS/SERVICES. The
  Default `CompanyFinancialPeriod` exposes the org field as `Company`
  (not OrganizationID/BranchID — those 500 "given key was not present");
  `?$filter=Company eq 'CAPITAL'` answers `200` with **0 rows**. The field
  is real and filterable, the view is simply empty without context. This
  is the decisive negative: the naive approach both the spec row and the
  first instinct reach for does not work.
- **Key-URL naming the org.** `/CompanyCalendar/<org>` → 500
  InvalidOperation (wrong key count — the key is a two-field tuple);
  `/CompanyCalendar/<org>/<year>` and `/<year>/<org>` → 500
  NoEntitySatisfies for every org × {2012, 2024, 2025}. The query runs but
  the view is empty; the key-URL carries no org context the list GET lacks.
- **Branch at login (per-org session).** Branch CDs pulled off the control
  plane (`sqlcmd` on the Branch table, CompanyID 4): CA, CAPITAL,
  PRODRETAIL, PRODWHOLE, SERVEAST, SERVWEST, TX. Login with a **valid**
  branch CD (`branch: "CAPITAL"`) → 500 "Invalid credentials", under both
  the `tenant` and the legacy `company` spelling of the tenant field. So
  the contract `/entity/auth/login` on this build (26.101.0225) does not
  give the `admin` user a branch-scoped session for the SalesDemo dataset —
  the SalesDemo dataset ships its own roles, and admin's branch access is
  not what a clean bootstrap-created tenant's is. Per-org login is not a
  reachable mechanism here.

**Conclusion.** Across both endpoints and every access path — list filter,
key-URL, per-org login — the GL201100 org-scoped views stay
unconditionally empty in a contract session. Single-org tenants
(COMPANY/LAB5) worked only because the sole organization is the session's
default org; there is no contract-API handle to select a different one.
The contract shape change T59 anticipated (add OrganizationID to
CompanyPeriod, bump the endpoint) would not help: OrganizationID as a
non-key field already exists on CompanyCalendar and filtering it returns
nothing, and as a key field the key-URL path already fails. The premise
that org context is settable from outside a sole-org default is not
supported by the live system. T59 as scoped cannot be built; the finding
routes to the spec (backprop / re-scope), not to code.

## Correction, same day: branch-login works for OUR admin — the block was dataset-specific

The "cannot be built" verdict above is scoped to the SalesDemo dataset,
not to the contract API. Re-tested on the `Company` tenant (id 2, ours,
bootstrap-configured, sole branch `LAB5`): a plain login reads
`CompanyCalendar=1 / CompanyPeriod=12` (sole org is the session default —
expected), and **`{tenant: Company, branch: LAB5}` logs in clean and
reads the same** — no "Invalid credentials". So our bootstrap-preset admin
(`-aun`/`-aup`/`-auc`, full rights) *can* hold a branch-scoped session; the
SalesDemo dataset's own `admin` roles were what refused branch selection.

That flips the direction. The org-context handle the contract API does
expose is **the branch at login** — a per-org session, "per-org iteration"
in its real form (iterate organizations, open a branch-scoped session per
org, read that org's calendar/periods). The residual unknown is only
whether branch-login *switches* context on a genuinely multi-org tenant
(the single-org test can confirm access but not selection) — and that gets
proven by the first multi-org config we build, not by more SalesDemo
archaeology.

Project decision (operator, this session): stop leaning on the SalesDemo
dataset for the multi-org story; develop a multi-org company from scratch
as config-as-code (multiple organizations + branches + per-org ledger
links + per-org calendars in the baseline), and make extract/diff org-aware
through per-org branch sessions. This re-scopes T59 (mechanism found, not a
dead end) and the SalesDemo-based e2e legs (T50/T58) onto our own multi-org
tenant. Shape it via design before building — it touches the client session
model (a branch dimension), the baseline's org structure, and the
extract/diff iteration.

**Friction: API Login Limit.** The probing tripped `500 API Login Limit`
(PX.LicensePolicy) — the license caps concurrent API sessions, and this
build's cap is low with a slow post-logout release (three quick logins
re-trip it even after a clear). Each `login` counts until logout or idle
timeout; the tenant guard's sign-in-page GET adds to the churn. Clearing
it does **not** need a host reboot: `Restart-WebAppPool -Name
'AcumaticaERP'` (the same recycle `acu tenant delete` runs, one SSH call)
resets the in-memory session counter instantly and warms the app on the
next request. Waiting out the idle timeout is the no-touch alternative.
Probe scripts must stay session-frugal: one login, many GETs.
