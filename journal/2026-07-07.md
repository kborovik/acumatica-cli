# 2026-07-07

## Skeleton verified end-to-end; snapshot plan dead; bootstrap blocker found

**Built.** Repo scaffolded as a uv Python project: `acu` CLI with
`tenant list|create|delete` (ac.exe `CompanyConfig` over SSH), `apply` (baseline
YAML → REST PUT upserts), `diff` (drift check, exit 1 on drift); Makefile front
door; encrypted `.env`; verified references in `docs/ac-exe.md` and
`docs/rest-api.md`. All against acu-dev1 (26.101.0225).

**Proven.** `baseline/uoms.yaml` round-trips: exported from the configured tenant,
`acu apply` is idempotent (re-run changes nothing), `acu diff` reports zero drift.
The core apply/diff loop works end-to-end.

**Dead end — snapshots.** The plan assumed "create tenant, then load a snapshot
via a CLI flag." Verified false: neither the 26.101 binary help (`ac.exe -?`) nor
the current parameter docs expose any snapshot operation, and the old
`PXInstanceHelper` API is gone from current builds. Snapshots exist only on the
Tenants screen (SM203520); community reports automating that screen's actions is
unreliable. Rescoped: the baseline lives entirely in reference-data-as-code.
Better outcome anyway — the whole baseline is diffable text, no binary blob in
git.

**Blocker — clean-tenant bootstrap.** On a fresh tenant (`CompanyType=''`), most
entity GETs/PUTs fail with HTTP 500 `PXSetupNotEnteredException: The required
configuration data is not entered on the Company Branches form`. `FinancialYear`
and `UnitsOfMeasure` respond; `Currency` doesn't; `CompaniesStructure` GET fails
with a BqlDelegate error (PUT unverified). Consequence: seeding needs an ordered
bootstrap step (enable features, create company/branch) before reference data can
flow. **Next experiment:** does `PUT CompaniesStructure` work on a clean tenant,
and can feature enablement (CS100000) be scripted at all?

**Gap — payment terms.** The Default 25.200.001 endpoint has no Credit Terms
entity (`ShippingTerm` is a different thing). Plan: publish a small custom
endpoint extension — itself version-controlled, which reinforces the thesis.

**Errors & solutions:**

- `ac.exe -cm:CompanyConfig` defaults `-dbnew` to `True` (= create a new
  database) — always pass `-dbnew:"False"` when operating on an existing DB.
- Official docs mislabel `CompanyType` as a boolean; working values are dataset
  names (`''` = clean, `SalesDemo` = demo). Verified via the infra deployment.
- REST logout via curl returns `411 Length Required` unless you send an explicit
  `Content-Length: 0` header.
- Always `POST /entity/auth/logout` — sessions count against the license's
  concurrent API user cap; leaked sessions exhaust a trial instance fast.
- Pass `tenant` on login once more than one tenant is visible — it selects which
  tenant the session operates on.

**Next.** (1) Bootstrap experiment on a scratch tenant — the critical-path
unknown. (2) Expand `baseline/` in dependency order: currencies → financial
calendar → chart of accounts/ledger → tax categories/zones → customer/vendor/item
classes. (3) Payment-terms endpoint extension. (4) `acu provision` one-command
orchestration (create → bootstrap → apply → diff). (5) Drift proof and timings
last, against the final pipeline.

## Evening — bootstrap experiment: no API-only path; CustomizationApi is the route

**Setup.** Created scratch tenant 3 (`CompanyType=''`) with
`acu tenant create`. Creation takes ~40 s, not minutes — after fixing a new
ac.exe gotcha (below).

**Surprise #1 — the "configured" tenant isn't.** The entity probe against
tenant 2 (`Company`) fails exactly like a virgin tenant. The instance has
never been through initial setup — `FeaturesSet` is empty for every tenant.
Everything this bootstrap builds is needed for the *primary* tenant too, not
just scratch copies; the uoms.yaml proof worked only because the UOM screen
has no setup dependency. (The pain is even more universal than assumed.)

**Surprise #2 — the `tenant` login parameter is a silent foot-gun.** REST
login returns 204 for *any* tenant name — including `DefinitelyBogus` — and
the session silently lands on the default tenant. Proof: marker UOM rows
written under four different tenant names all landed on CompanyID 2 (checked
in the DB, then cleaned up). Worse: the running site doesn't even know the
new tenant exists — the sign-in page doesn't list it; the tenant map loads
at app start. So every "scratch tenant" probe tonight actually ran against
tenant 2 — which is fine for the findings (tenant 2 is equally unconfigured,
confirmed explicitly), but the lesson is brutal: **without a post-login
tenant guard, config-as-code can silently write to the wrong tenant.**
`acu apply`/`diff` must grow that guard. Presumed fix for visibility: recycle
the `AcumaticaERP` app pool after tenant create — unverified, the restart
needs operator approval (denied for the agent by the permission classifier —
correctly, it's a state-changing service restart).

**Result — the API cannot bootstrap a tenant by itself:**

- `PUT CompaniesStructure`: 500 `Operation failed` on every variant tried —
  the one org-structure entity in the Default endpoint cannot create a
  company/branch. GET fails too (`BqlDelegate in view Filter`).
- Failures split into two clean classes (full map in `docs/rest-api.md`):
  500 `PXSetupNotEnteredException` naming the missing setup form (the
  dependency chain: Companies → Company Branches → Branch → GL Preferences →
  Financial Year) and 403 insufficient-rights on feature-gated screens
  (AR/AP/IN classes, GL subaccounts — features never enabled).
- Login is NOT a blocker: `admin` + the instance admin password works on a
  virgin tenant over REST — no first-login password-change dance.
  *(Corrected 2026-07-08: this was an artifact — the probes hit tenant 2,
  long past first login. Fresh tenants do hit the first-login wall.)*
- Insert datasets are no shortcut: the box ships `SalesDemo`/`System`/
  `T100`/`U100`; the training sets carry transactional data, not a clean
  baseline.
- The **CustomizationApi works on a virgin tenant**
  (`POST /CustomizationApi/getPublished` → 200).

**Decision — the bootstrap route:** tenant create (ac.exe) → publish a small
bootstrap customization package via the CustomizationApi → its custom
endpoint exposes Companies (CS101500), Enable/Disable Features (CS100000),
and Credit Terms (CS206500 — closes the payment-terms gap with the same
stone) → enable features + create company/branch over REST → seed `baseline/`
via the Default endpoint. The package is endpoint XML in a zip:
version-controlled, no C#. Open risk: whether CS100000 accepts
endpoint-driven writes; fallback is a `CustomizationPlugin` that flips
`FeaturesSet` on publish (per-tenant, but ships C#). Prior art: the
[Customization API](https://community.acumatica.com/develop-customizations-288/customization-api-17892)
thread and the
[feature-flag-on-publish](https://community.acumatica.com/develop-customizations-288/initial-value-of-feature-flag-24036)
pattern.

**Errors & solutions:**

- `ac.exe -cm:CompanyConfig` on 26.101 requires `-iname:"<instance>"`;
  without it: `Site with name '' doesn't exist in the system`. The official
  docs' CompanyConfig example omits the flag. Fixed: `instances/*.yaml` now
  carries `instance_name` and the CLI passes it.
- `LoginName=Scratch` in the `-company` parameter did not take — the tenant
  got auto-named `Company2` (no LoginName column exists in `dbo.Company`;
  where that sub-key lands is unknown). *(Corrected 2026-07-08: it did take —
  `LoginName` lands in `dbo.Company.CompanyKey`, the actual sign-in name;
  `CompanyCD` is just an auto-generated internal code.)*

**Next.** (1) Recycle the app pool (operator) and verify tenant 3 becomes
selectable — then re-run the probe against the real scratch tenant and add
the recycle + a post-login tenant guard to the CLI. (2) Author the bootstrap
package (build the endpoint once in the UI, export, commit the XML); prove
features + company creation on the scratch tenant. (3) Wire `acu provision`.
Scratch tenant 3 (`Company2`) stays in place as the guinea pig.
