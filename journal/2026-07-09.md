# 2026-07-09

## T17: exit-code propagation centralized in `_ssh`

Closed the last open SPEC task. Follow-through from yesterday's B4 bug
(the sqlcmd tenant-list call over SSH silently returned success on
failure because PowerShell-over-ssh reports exit 0 for a failed native
command): instead of remembering to hand-append `exit $LASTEXITCODE` at
every call site, `_ssh` in `tenant.py` now appends it itself, once, as
the single choke point (V18). The two call sites that had the suffix
hand-appended (`recycle_app_pool`, `_company_config`) lost it; the
sqlcmd read in `list()` — the one B4 caught missing — is now covered
automatically.

New test pins the invariant from both directions: every remote command
recorded by the fake SSH runner must end with the suffix and contain it
exactly once. A missing suffix means a swallowed failure; a doubled one
means a call site regressed to hand-appending.

`make check` green (88 tests). §T backlog is now fully closed — next
work needs a new spec row (baseline expansion in dependency order is
the leading open milestone).

## T18: the V9 ASCII audit becomes a script

The check-extras §V.9 recipe was a grep plus an eye-applied exemption
filter ("docstrings and comments don't count") — a judgment call repeated
on every audit run. `scripts/check-ascii` mechanizes the filter: `.py`
files go through `tokenize` (COMMENT tokens skipped; docstring STRING
tokens skipped via `ast` first-statement spans, so a trailing comment is
exempt while code on the same line still flags), `.cs` exempts `//`
comment lines, `.xml` exempts `<!-- -->` regions across lines. Directory
arguments recurse over those three suffixes.

One self-referential wrinkle: the report format itself must obey the
invariant it enforces — printing the offending glyph would put non-ASCII
on stdout. Violations emit as `file:line: U+XXXX` (first offending
codepoint) instead, and a test pins the report as ASCII-encodable.

The recipe command in `.claude/check-extras.md` flipped to
`scripts/check-ascii src/` in the same commit, and a test runs it against
the real `src/` tree so the current em-dash-in-comments population stays
pinned as exempt. `make check` green (98 tests).

## T19: the V1/V10/V18 drift greps become a check hook

Same mechanization move as T18, applied to the three invariants /sdd:check
re-derived as ad-hoc greps every run. `.claude/scripts/check-extras.sh`
now owns them and speaks the check extras-hook contract — bare
`id|verdict|evidence` rows on stdout, exit 1 on any VIOLATE:

- **V1 plane split** — `tenant.py` (control plane) must not mention
  `httpx`; `client.py` (data plane) must not mention `subprocess`.
- **V10 model inheritance** — outside `models.py`, no class subclasses
  `BaseModel` directly; models route through `models.Model`. The SPEC
  row's telegraph ("`^class` in src/ must inherit Model") reads literally
  as "every class is a model", which would flag `AcumaticaClient`,
  `TenantManager`, and the HTMLParser login form — none of them models,
  all HOLD today. The direct-BaseModel grep is the only mechanical
  reading consistent with V17's satisfiable-gate rule, so that is what
  shipped.
- **V18 choke point** — `exit $LASTEXITCODE` appears exactly once in
  `src/`, and an awk pass confirms the enclosing def is `_ssh` (a lone
  suffix in the wrong function would otherwise pass the count check).

VIOLATE branches can't be pinned against the real tree, so the tests
copy the hook into a synthetic repo under `tmp_path` (the script resolves
its root from its own location) and break each invariant there: httpx in
tenant.py, a stray `BaseModel` subclass, a hand-appended call-site
suffix, a deleted suffix, and a suffix living in the wrong def. The
check-extras recipe rows landed in the same commit, as did versioning
`.claude/.gitignore` (check-state.json stays an unversioned cache).
`make check` green (106 tests). §T backlog closed again.

## make e2e: the live provision lifecycle becomes a test tier

Until now the only automated proof of the whole point of the tool — SPEC
goal, "run one command, get a byte-identical configured tenant" — was a
manual ritual in the data repo (`make decrypt && make diff`), and neither
of the two live-only bug classes in the ledger (B1's provision gate, B2's
`projectContentBase64` binder mismatch) could ever be caught by the
offline suite. `make e2e` closes that gap without touching the offline
commit gate.

Design: a new `tests/e2e/test_provision_lifecycle.py` carries a pytest
`e2e` marker that the default suite deselects through `addopts`
(`-m "not e2e"` — the command line's `-m e2e` wins because the last `-m`
takes precedence), so `make check` still runs 106 offline tests and never
opens a socket. The e2e tests drive the installed `acu` binary through
`subprocess`, not Click's `CliRunner`, because the thing worth proving
live is the machine interface itself: exit 0 on clean, exit 2 on drift,
status lines on stderr.

The lifecycle is one session bracketed by a fixture: clear any leftover
`E2E` tenant from a crashed run, pick the next free CompanyID, then (1)
`acu provision --login E2E` must exit 0 ending in "no drift", (2) an
independent `acu -t E2E diff baseline` must agree, (3) a second provision
must exit 0 through the skip paths ("skipping create" — the V4
resume-verifies-state rule observed live), (4) a copy of a baseline YAML
with one mutated field must make `diff` exit 2 with a DRIFT report. The
fixture teardown always deletes the tenant and recycles the pool
(`TenantManager.delete` has no confirmation prompt, unlike the CLI), so
nothing persists on acu-dev1 between runs.

Two discoveries along the way. First, everything lives in one test module
rather than a conftest: `tests/e2e/` has no `__init__.py`, so a test
module cannot import types from its own conftest, and a single file
dissolves the problem. Second, this repo's data symlinks were missing
`bootstrap` — `data_root()` resolves to wherever `acu.yaml` is found, so
a provision run from this repo would have silently skipped bootstrap YAML
seeding while the same command from the data repo seeded it. The symlink
(and a `.gitignore` line) landed with the tier; without the e2e work the
asymmetry would have kept lurking.

First full run: 4 passed in 269 seconds, tenant list clean afterwards.
The wall time is almost entirely the three app-pool recycles plus the
customization publish, which is the honest cost of the real pipeline.

Follow-up: the first version of the tier was silent for the whole 4.5
minutes — `run_acu` captured everything and pytest captured the rest, so
a stuck run and a working run looked identical. Fixed by streaming: the
runner now pumps the child's stdout/stderr to the terminal line by line
through reader threads while still buffering both for the assertions,
and `make e2e` passes `-o addopts= -m e2e -v -s` so pytest neither
deselects nor swallows the stream. Progress is now acu's own step lines
plus one `$ acu ...` echo per invocation.

The verification re-run also delivered an accidental live lesson: a
manual tenant delete happened on the box mid-run, and the tier caught
the collision in two distinct shapes. The concurrent ac.exe run made
create fail with "The Database 'AcumaticaDB' has been locked by another
update process" (transient — cleared by the next test), and the manual
delete without a recycle left the stale tenant map, so `acu -t E2E diff`
logged in against the *default* tenant and reported a clean, false "no
drift" — friction #11 observed end to end, this time through the diff
path. The client's tenant guard refuses an empty tenant but cannot know
a named tenant silently rerouted; only the recycle discipline (V5)
protects that. A clean re-run after the box settled: 4 passed in 360
seconds, tenant list clean.

## T21: the landed-tenant guard — archaeology, a myth corrected, and a probe

Yesterday's B5 left a hole no discipline can close from the outside: a
named REST login can silently land on the wrong tenant, and nothing in
the client noticed. T21's brief was to find a verified way to ask the
server "which tenant did this session actually land on?" and refuse the
session on a mismatch, inside `AcumaticaClient.__enter__`.

The archaeology took most of the work, and it corrected a documented
belief along the way. The docs said tenant routing is strict once the
map is fresh (unknown name → 500 `PXUndefinedCompanyException`). That
turns out to be true only on a multi-tenant instance. With a single
login-able tenant in the database, login validates nothing: any tenant
value — a deleted tenant's name, pure nonsense — returns 204 and lands
on the only tenant there is. Verified by deleting the scratch tenant,
recycling properly, waiting out the warmup, and logging in as the dead
name anyway. So the "strict" behavior the docs promised only ever
existed when there were two or more tenants to choose between, and the
false-green `diff` from B5 does not even need a stale map on a
single-tenant box — it is the steady state. A client-side guard is not
defense-in-depth; it is the only defense.

Finding the probe was a process of elimination. The login response
carries nothing (204, no tenant cookie on this build — `UserBranch` is
empty, `Locale` is tenant-neutral). `GET /entity` and
`getPublished` are landing-invariant. DAC-level OData is 404 on this
build. The `/Frames` page `<title>` names the landed tenant — but in
`CompanyCD` space (`Company2` for a tenant whose login name is
`T21Probe`), and REST login speaks login-name space, so the two never
meet. The HTML5 shell's company markup and the `screenLink`
`CompanyID` query parameter both looked right — login-name space,
reflects the landing, survives the stale-map reroute — and both
evaporate the moment the database drops to one tenant (the gate is a
live tenant count, not the map: the parameter vanished immediately
after the delete, before any recycle). The one surface that names the
landed tenant by login name in every observed state — multi-tenant
fresh map, mid-reroute under a stale map, single-tenant — is an
authenticated GET of the sign-in page itself: `/Frames/Login.aspx`
renders a hidden `txtSingleCompany` input whose value is the session's
landed tenant, and the GET does not disturb the session. That is now
the guard: parse it after login, compare case-insensitively against
the requested tenant, log out and refuse on mismatch (logout must be
explicit — `__exit__` never runs when `__enter__` raises). If the page
shape ever changes on an upgrade, the guard fails closed rather than
degrading into the old blindness.

Two smaller live lessons. Reproducing B5 deliberately (delete without
recycle) confirmed the reroute window end to end, with the probe
naming `Company` for a session that asked for `T21Probe`. And
`Restart-WebAppPool` returns before the new worker serves — IIS
overlapped recycle means probes fired right after a recycle still hit
the old worker and its old map for tens of seconds, which briefly made
the fresh map look stale until the warmup wait went in.

The e2e tier gained the B5 regression: `acu -t NoSuchTenantB5 diff
baseline` must exit 1, whichever path fails it (login 500 on a
multi-tenant box, tenant guard on a single-tenant one). Run live: the
guard refused with the landed tenant named in the error. Offline, the
mock transports learned to serve the sign-in page fragment, and three
new tests pin the mismatch refusal, the logout-on-refusal, and the
fail-closed missing-marker path. `make check` green (109 tests).

## T22: global `--host` flag

Small surface, one sharp edge. The flag mirrors `-t`: a global before
the subcommand that overrides one acu.yaml value for a single run. But
where `-t` can be a `model_copy` after the Instance is built (tenant is
a leaf value), `host` cannot — `base_url` and `ssh` derive from it in
the model validator, so a post-hoc copy would swap the host while both
derived fields kept pointing at the old box. The override therefore
happens inside `load_instance`, on the raw config dict before the
Instance is constructed, so the swap flows through the same
`_derive_urls` path the file's own host does. That also gives the
precedence rule for free: an explicit acu.yaml `base_url` or `ssh`
still wins over the swapped host, exactly as it wins over the file's
host.

Four new tests: re-derivation of both URLs from the swapped host,
explicit-override precedence (with a deliberately different host so the
assertion cannot pass by coincidence), an end-to-end `acu --host ...
config show` through the real loader, and the V16 rejection of the flag
after the subcommand. `make check` green (113 tests). §T backlog is
fully closed again.

## T23: pip install acumatica-cli — trusted publishing to PyPI

The release pipeline grew its last leg: `make release <part>` already
bumped, tagged, and published a GitHub release; now the release event
triggers `.github/workflows/release.yml`, which builds the sdist and
wheel with `uv build` and publishes to pypi.org through OIDC trusted
publishing — no API token stored anywhere (V19). The workflow also
guards the V19 tag rule mechanically: a step asserts the pushed tag
equals `v` plus the pyproject version before building, so a hand-made
tag that drifts from the version fails the run instead of shipping a
mislabeled package.

The verification release (v0.2.2) earned its keep by failing twice.
PyPI answered the first publish with `invalid-publisher` — valid OIDC
token, no matching publisher. The claims GitHub sent were all correct,
which pointed the mismatch at the registration form, and the culprit
was the PyPI form's Repository name field: it takes the bare repository
name (`acumatica-cli`), but it had been filled with the
`owner/repo` form (`kborovik/acumatica-cli`) — the owner is its own
field, so PyPI was effectively looking for a repository named
`kborovik/acumatica-cli` under user `kborovik`. The failure mode is
quiet because the form accepts the slash without complaint. One field
edit and a rerun of the failed job later (no new release needed — the
release event had already fired), the publish went through with digital
attestations.

Verified end to end: `pip install acumatica-cli==0.2.2` from pypi.org
into a scratch venv, `acu --version` reports 0.2.2. The §T backlog is
fully closed.
