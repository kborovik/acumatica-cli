## T17: exit-code propagation centralized in `_ssh`

Closed the last open SPEC task. Follow-through from yesterday's B4 bug
(the sqlcmd tenant-list call over SSH silently returned success on
failure because PowerShell-over-ssh reports exit 0 for a failed native
command): instead of remembering to hand-append `exit $LASTEXITCODE` at
every call site, `_ssh` in `tenant.py` now appends it itself, once, as
the single choke point (V18). The two call sites that had the suffix
hand-appended (`recycle_app_pool`, `_company_config`) lost it; the
sqlcmd read in `list()` — the one B4 caught missing — is now covered
automatically.

New test pins the invariant from both directions: every remote command
recorded by the fake SSH runner must end with the suffix and contain it
exactly once. A missing suffix means a swallowed failure; a doubled one
means a call site regressed to hand-appending.

`make check` green (88 tests). §T backlog is now fully closed — next
work needs a new spec row (baseline expansion in dependency order is
the leading open milestone).

## T18: the V9 ASCII audit becomes a script

The check-extras §V.9 recipe was a grep plus an eye-applied exemption
filter ("docstrings and comments don't count") — a judgment call repeated
on every audit run. `scripts/check-ascii` mechanizes the filter: `.py`
files go through `tokenize` (COMMENT tokens skipped; docstring STRING
tokens skipped via `ast` first-statement spans, so a trailing comment is
exempt while code on the same line still flags), `.cs` exempts `//`
comment lines, `.xml` exempts `<!-- -->` regions across lines. Directory
arguments recurse over those three suffixes.

One self-referential wrinkle: the report format itself must obey the
invariant it enforces — printing the offending glyph would put non-ASCII
on stdout. Violations emit as `file:line: U+XXXX` (first offending
codepoint) instead, and a test pins the report as ASCII-encodable.

The recipe command in `.claude/check-extras.md` flipped to
`scripts/check-ascii src/` in the same commit, and a test runs it against
the real `src/` tree so the current em-dash-in-comments population stays
pinned as exempt. `make check` green (98 tests).

## make e2e: the live provision lifecycle becomes a test tier

Until now the only automated proof of the whole point of the tool — SPEC
goal, "run one command, get a byte-identical configured tenant" — was a
manual ritual in the data repo (`make decrypt && make diff`), and neither
of the two live-only bug classes in the ledger (B1's provision gate, B2's
`projectContentBase64` binder mismatch) could ever be caught by the
offline suite. `make e2e` closes that gap without touching the offline
commit gate.

Design: a new `tests/e2e/test_provision_lifecycle.py` carries a pytest
`e2e` marker that the default suite deselects through `addopts`
(`-m "not e2e"` — the command line's `-m e2e` wins because the last `-m`
takes precedence), so `make check` still runs 106 offline tests and never
opens a socket. The e2e tests drive the installed `acu` binary through
`subprocess`, not Click's `CliRunner`, because the thing worth proving
live is the machine interface itself: exit 0 on clean, exit 2 on drift,
status lines on stderr.

The lifecycle is one session bracketed by a fixture: clear any leftover
`E2E` tenant from a crashed run, pick the next free CompanyID, then (1)
`acu provision --login E2E` must exit 0 ending in "no drift", (2) an
independent `acu -t E2E diff baseline` must agree, (3) a second provision
must exit 0 through the skip paths ("skipping create" — the V4
resume-verifies-state rule observed live), (4) a copy of a baseline YAML
with one mutated field must make `diff` exit 2 with a DRIFT report. The
fixture teardown always deletes the tenant and recycles the pool
(`TenantManager.delete` has no confirmation prompt, unlike the CLI), so
nothing persists on acu-dev1 between runs.

Two discoveries along the way. First, everything lives in one test module
rather than a conftest: `tests/e2e/` has no `__init__.py`, so a test
module cannot import types from its own conftest, and a single file
dissolves the problem. Second, this repo's data symlinks were missing
`bootstrap` — `data_root()` resolves to wherever `acu.yaml` is found, so
a provision run from this repo would have silently skipped bootstrap YAML
seeding while the same command from the data repo seeded it. The symlink
(and a `.gitignore` line) landed with the tier; without the e2e work the
asymmetry would have kept lurking.

First full run: 4 passed in 269 seconds, tenant list clean afterwards.
The wall time is almost entirely the three app-pool recycles plus the
customization publish, which is the honest cost of the real pipeline.

Follow-up: the first version of the tier was silent for the whole 4.5
minutes — `run_acu` captured everything and pytest captured the rest, so
a stuck run and a working run looked identical. Fixed by streaming: the
runner now pumps the child's stdout/stderr to the terminal line by line
through reader threads while still buffering both for the assertions,
and `make e2e` passes `-o addopts= -m e2e -v -s` so pytest neither
deselects nor swallows the stream. Progress is now acu's own step lines
plus one `$ acu ...` echo per invocation.

The verification re-run also delivered an accidental live lesson: a
manual tenant delete happened on the box mid-run, and the tier caught
the collision in two distinct shapes. The concurrent ac.exe run made
create fail with "The Database 'AcumaticaDB' has been locked by another
update process" (transient — cleared by the next test), and the manual
delete without a recycle left the stale tenant map, so `acu -t E2E diff`
logged in against the *default* tenant and reported a clean, false "no
drift" — friction #11 observed end to end, this time through the diff
path. The client's tenant guard refuses an empty tenant but cannot know
a named tenant silently rerouted; only the recycle discipline (V5)
protects that. A clean re-run after the box settled: 4 passed in 360
seconds, tenant list clean.

## T21: the landed-tenant guard — archaeology, a myth corrected, and a probe

Yesterday's B5 left a hole no discipline can close from the outside: a
named REST login can silently land on the wrong tenant, and nothing in
the client noticed. T21's brief was to find a verified way to ask the
server "which tenant did this session actually land on?" and refuse the
session on a mismatch, inside `AcumaticaClient.__enter__`.

The archaeology took most of the work, and it corrected a documented
belief along the way. The docs said tenant routing is strict once the
map is fresh (unknown name → 500 `PXUndefinedCompanyException`). That
turns out to be true only on a multi-tenant instance. With a single
login-able tenant in the database, login validates nothing: any tenant
value — a deleted tenant's name, pure nonsense — returns 204 and lands
on the only tenant there is. Verified by deleting the scratch tenant,
recycling properly, waiting out the warmup, and logging in as the dead
name anyway. So the "strict" behavior the docs promised only ever
existed when there were two or more tenants to choose between, and the
false-green `diff` from B5 does not even need a stale map on a
single-tenant box — it is the steady state. A client-side guard is not
defense-in-depth; it is the only defense.

Finding the probe was a process of elimination. The login response
carries nothing (204, no tenant cookie on this build — `UserBranch` is
empty, `Locale` is tenant-neutral). `GET /entity` and
`getPublished` are landing-invariant. DAC-level OData is 404 on this
build. The `/Frames` page `<title>` names the landed tenant — but in
`CompanyCD` space (`Company2` for a tenant whose login name is
`T21Probe`), and REST login speaks login-name space, so the two never
meet. The HTML5 shell's company markup and the `screenLink`
`CompanyID` query parameter both looked right — login-name space,
reflects the landing, survives the stale-map reroute — and both
evaporate the moment the database drops to one tenant (the gate is a
live tenant count, not the map: the parameter vanished immediately
after the delete, before any recycle). The one surface that names the
landed tenant by login name in every observed state — multi-tenant
fresh map, mid-reroute under a stale map, single-tenant — is an
authenticated GET of the sign-in page itself: `/Frames/Login.aspx`
renders a hidden `txtSingleCompany` input whose value is the session's
landed tenant, and the GET does not disturb the session. That is now
the guard: parse it after login, compare case-insensitively against
the requested tenant, log out and refuse on mismatch (logout must be
explicit — `__exit__` never runs when `__enter__` raises). If the page
shape ever changes on an upgrade, the guard fails closed rather than
degrading into the old blindness.

Two smaller live lessons. Reproducing B5 deliberately (delete without
recycle) confirmed the reroute window end to end, with the probe
naming `Company` for a session that asked for `T21Probe`. And
`Restart-WebAppPool` returns before the new worker serves — IIS
overlapped recycle means probes fired right after a recycle still hit
the old worker and its old map for tens of seconds, which briefly made
the fresh map look stale until the warmup wait went in.

The e2e tier gained the B5 regression: `acu -t NoSuchTenantB5 diff
baseline` must exit 1, whichever path fails it (login 500 on a
multi-tenant box, tenant guard on a single-tenant one). Run live: the
guard refused with the landed tenant named in the error. Offline, the
mock transports learned to serve the sign-in page fragment, and three
new tests pin the mismatch refusal, the logout-on-refusal, and the
fail-closed missing-marker path. `make check` green (109 tests).

## T22: global `--host` flag

Small surface, one sharp edge. The flag mirrors `-t`: a global before
the subcommand that overrides one acu.yaml value for a single run. But
where `-t` can be a `model_copy` after the Instance is built (tenant is
a leaf value), `host` cannot — `base_url` and `ssh` derive from it in
the model validator, so a post-hoc copy would swap the host while both
derived fields kept pointing at the old box. The override therefore
happens inside `load_instance`, on the raw config dict before the
Instance is constructed, so the swap flows through the same
`_derive_urls` path the file's own host does. That also gives the
precedence rule for free: an explicit acu.yaml `base_url` or `ssh`
still wins over the swapped host, exactly as it wins over the file's
host.

Four new tests: re-derivation of both URLs from the swapped host,
explicit-override precedence (with a deliberately different host so the
assertion cannot pass by coincidence), an end-to-end `acu --host ...
config show` through the real loader, and the V16 rejection of the flag
after the subcommand. `make check` green (113 tests). §T backlog is
fully closed again.

## T23: pip install acumatica-cli — trusted publishing to PyPI

The release pipeline grew its last leg: `make release <part>` already
bumped, tagged, and published a GitHub release; now the release event
triggers `.github/workflows/release.yml`, which builds the sdist and
wheel with `uv build` and publishes to pypi.org through OIDC trusted
publishing — no API token stored anywhere (V19). The workflow also
guards the V19 tag rule mechanically: a step asserts the pushed tag
equals `v` plus the pyproject version before building, so a hand-made
tag that drifts from the version fails the run instead of shipping a
mislabeled package.

The verification release (v0.2.2) earned its keep by failing twice.
PyPI answered the first publish with `invalid-publisher` — valid OIDC
token, no matching publisher. The claims GitHub sent were all correct,
which pointed the mismatch at the registration form, and the culprit
was the PyPI form's Repository name field: it takes the bare repository
name (`acumatica-cli`), but it had been filled with the
`owner/repo` form (`kborovik/acumatica-cli`) — the owner is its own
field, so PyPI was effectively looking for a repository named
`kborovik/acumatica-cli` under user `kborovik`. The failure mode is
quiet because the form accepts the slash without complaint. One field
edit and a rerun of the failed job later (no new release needed — the
release event had already fired), the publish went through with digital
attestations.

Verified end to end: `pip install acumatica-cli==0.2.2` from pypi.org
into a scratch venv, `acu --version` reports 0.2.2. The §T backlog is
fully closed.

## ac.exe archaeology: the config-file format, and hidden export/import verbs

Two questions from the operator — "what does the native configuration
dump look like?" and "could we dump a sales company to XML, version it,
edit it, upload it?" — turned into a decompilation session that
recovered two undocumented ac.exe surfaces. Both are now written up in
`docs/ac-exe.md`.

First, the `-f` configuration file. No example exists anywhere — not on
the box, not in the official help (checked both wiki pages), and the
console binary has no save flag; only the GUI wizard's finish page
writes one. String-mining the 4 MB exe found the loader's error
templates (which is how the `<Root>` root element and the strict
node/attribute validation surfaced via probe files), but the real
answer came from pulling the binary local and decompiling it with
ilspycmd: `ConsoleOptionExtensions.SaveToFile`/`LoadFromFile` define
the whole format — one element per option named by its long CLI name,
value in a `Value` attribute, `Description` decorative, nulls omitted,
unknowns ignored, tenants under `<company>`, and two write-only
elements embedding the equivalent command line. Then the neat trick:
loading ac.exe (net48) into remote PowerShell via reflection and
calling the wizard's own `SaveToFile` produced a genuine dump — after
seeding option defaults the way `LoadFromFile` does and installing a
`ConfigContext` into the static `SysData.Context` (the command-line
builder at the tail of `SaveToFile` reads global wizard state and
NREs otherwise). Passwords land in the file in clear text. Key scoping
fact: the file is installer parameters only — DB/IIS/instance/tenant
list — no in-tenant ERP data, so it is not a config-as-code surface.

Second, and bigger: `ConfigStart.opcodeHandlers` dispatches a whole
undocumented verb family when argv[0] is a bare word — `database`,
`export`, `import`, `xmlentity`, `delta`, `util`, `mobilesitemap`.
`export xml 'mssql://Server=(local);Database=AcumaticaDB;Integrated
Security=SSPI?companyid=2' <folder>` ran live on acu-dev1: exit 0, one
XML per table that has rows for the tenant, and the format is exactly
the shipped-dataset format under `Database\Data\` — schema header plus
attribute-encoded rows. So `SalesDemo` is literally an `ac.exe export
xml` dump, and `CompanyType` at tenant creation is its import path;
there is also a direct `import xml <folder> <url>` verb (unverified —
it mutates). The operator's proposed loop (dump → version → edit →
upload) is therefore mechanically possible, but it is a raw-table
surface: cross-file integer ID webs, no business logic on import,
schema coupled to the build. Verdict for this project unchanged —
REST seeding stays the primary mechanism (it survives upgrades and
runs validations); the export verb earns a place as a potential
whole-tenant diff/verification and disaster-recovery tool, maybe a
future `acu` subcommand.

One process note: this session hand-rolled the utf-16le/base64
`-EncodedCommand` dance again before remembering `scripts/ps-remote`
exists (friction #33) — the helper works and should be the reflex.

## T24: the feature set becomes data — and the replay finds the next wall

B6's forensics said the bootstrap plugin's six hardcoded feature names
were config masquerading as tool source, and T24 makes them data: the
C# plugin now ships with an empty `Enabled` set holding only an
`/*ACU_FEATURES*/` sentinel, and `package_zip()` splices the names in
at package-build time from the data repo's `bootstrap/features.yaml`
(absent file falls back to the built-in six, now a Python code default
in `bootstrap.DEFAULT_FEATURES` — same layering as the `Instance`
defaults). A companion guard travels inside the plugin: any enabled
name that matches no `bool?` property on `FeaturesSet` gets called out
in the publish log, so a typo in features.yaml can no longer silently
enable nothing. One integration wrinkle: `features.yaml` lives in
`bootstrap/`, which provision sweeps as seed YAML — `expand_files` now
skips the name everywhere a directory expands (it is package-build
config, not an entity/records file), and a bootstrap directory whose
only YAML is features.yaml errors as having no seed files.

The data repo gained the eight-name list (six + `Multicurrency` +
`SubAccount`) and the live verify replayed the SalesDemo extract onto
a fresh tenant. The feature half of B6 is confirmed dead: FeaturesSet
row for the scratch tenant shows `Multicurrency=1, SubAccount=1`
(yesterday's six-feature tenant reads 0/0 beside it), all 15
subaccounts applied where B6 saw 403, and 161 accounts applied where
B6 saw 500.

But the replay found the next wall, and it rewrote half of B6's
attribution. The EUR-denominated accounts (5 of 166) still fail —
now as an orderly 422 "Currency 'EUR' cannot be found" instead of a
500 — because the Default endpoint's `Currency` entity is the
CM201000 *currency list* (Active/Symbol/DecimalPrecision/
UseForAccounting and nothing else, per the live schema dump). Setting
`UseForAccounting: true` there creates no CM202000 financial-currency
row, and GL accounts validate against the financial-currency table:
SalesDemo carries five CM.Currency rows, a fresh tenant gets base USD
only. So B6's "Account PUT 500 (Multicurrency off)" was half right —
the feature bit was one cause, the missing financial-currency row the
other, and only the feature bit was T24's to fix. Backpropped as B8
with the §C gap list extended (financial currency joins payment terms
as "no Default-endpoint entity") and T29 queued to front CM202000
from the Bootstrap custom endpoint, the same route that closed the
credit-terms gap.

Two smaller extract lessons from finishing the replay: 33 of the
extracted accounts referenced `AccountGroup`, which is PM201000 —
Projects-feature-gated AND absent from the extract — so the replay
strips the field exactly as the original extract stripped
`AccountClass` (the extract's own header documents that precedent);
and `ChartOfAccountsOrder` is server-derived (source said 4, the
fresh tenant computed 3 on three accounts), so extracts that carry
derived fields will always diff dirty — an extract-methodology note
for whenever the SalesDemo material graduates into `baseline/`.

## T25: the publish skip gate learns to read content, not existence

Friction #32 (B7 in the ledger) has been waiting since the day the
B3 fix landed: the publish skip gate required getPublished to list the
package AND getProject to find its content, which catches a recreated
tenant but still skips when the content is simply *different* — a
changed features.yaml, an edited plugin, or an older tool version's
package all sail through as "already published" and silently starve
the tenant of the current config. T25 makes the gate content-aware:
`package_zip()`'s output gets digested (sha256 over the project.xml
bytes — the XML serialization is deterministic, the zip container is
not) and the digest rides into the import call's `projectDescription`.
The skip now requires the published package's description to carry the
digest of the package we would build *now*; any mismatch — including
the digest-less descriptions of packages published before this gate
existed — falls through to a full reimport + republish carrying the
new digest.

The readback channel needed live archaeology, because the
CustomizationApi offers no obvious "get description" call. A read-only
probe against 26.101.0225 settled it: getPublished rows carry only
`name` (plus an `items` list of project-item keys), getProject's JSON
body is just `{hasConflicts, log, projectContentBase64}` — but the
content it returns is the *server's own re-serialization* of the
project, and its project.xml root carries the import's description in
the `description` attribute. So the one round-trip channel is: import
`projectDescription` in, root attribute of the re-served project.xml
out. The probe also caught the dev instance's default tenant sitting
in the exact B3 state (publication listed, content gone) while the
T24 scratch tenant `FromCode` held the real package — a live reminder
of why existence-only gates lie. Client/bootstrap split stays clean:
`client.py` grew `customization_project_content()` (transport, bytes
or None), and all package semantics — digest, description format,
root-attribute parse — live in `bootstrap.py`. Offline tests pin the
mismatch-to-republish path both ways (different-features digest and
legacy digest-less description), with the mock now serving a
server-style re-serialization instead of a `"UEs="` stub.

## T26: `acu config init` — the data repo scaffolds itself

Until now the only way to start a data repo was to copy
`acumatica-baseline` by hand and guess which files matter. T26 gives
the CLI a front door: `acu config init [--host <host>] [<dir>]` writes
a seven-file starter set — `acu.yaml` (host only, with a header comment
pointing at `acu config show`), `.env` with placeholder credentials,
a `.gitignore` covering `.env` and `schemas/`, a two-record
`baseline/uoms.yaml`, `bootstrap/company.yaml` and
`bootstrap/credit-terms.yaml` as per-file `endpoint:` override
examples, and `bootstrap/features.yaml` spelling out the built-in six.
The templates ship as package data (read via `importlib.resources`,
same route as the bootstrap plugin sources), every file is
skip-if-exists (`skip <file> (exists)`, exit 0, never overwrites),
and there is deliberately no `git init` and no gpg — version control
and secret encryption stay the operator's call.

The structural change underneath: the top-level Click group used to
resolve `load_instance()` eagerly for every subcommand, which would
have made `config init` die on the very discovery error it exists to
cure (no `acu.yaml` anywhere up the tree). Resolution now lives in a
`pass_instance` decorator that runs at command time — the group
callback just stashes the `-t`/`--host` globals — so init is simply a
command that never resolves, and every other command behaves
identically.

One near-miss worth recording: the first commit silently dropped five
of the seven templates. The repo's `.gitignore` lists `baseline`,
`bootstrap`, and `acu.yaml` for the dev symlinks, and unanchored
patterns match at any depth — so `src/acumatica_cli/templates/acu.yaml`
and both template subdirectories were ignored while the offline suite
passed green from the working tree. A fresh clone would have shipped a
broken `config init`. The patterns are now root-anchored (`/baseline`,
`/bootstrap`, `/acu.yaml`, `/.env`), and the wheel listing was checked
by hand (`uv build` + `unzip -l`) to confirm all seven templates ride
along. The lesson generalizes: tests that read package data from an
editable install cannot see what git excludes.

## T27: `acu config check` — the preflight that names what is broken

`config init` gives a newcomer a data repo; `config check` now tells
them whether it actually works, before any command tries to write
anything. Four probes in dependency order, one `ok`/`fail` line each on
stdout: discovery (the acu.yaml walk-up, parse, and host key), secrets
(`.env` plus `ACU_PASSWORD` — the value itself never printed), then the
two live planes probed independently — REST and ssh. A discovery or
secrets failure stops the run (nothing downstream can work without
them), but a REST failure still probes ssh and the other way around, so
one run names every broken layer instead of the first one only. Exit 0
when everything passes, 1 on any failure; 2 stays reserved for drift.

The REST probe cost nothing new: entering `AcumaticaClient` already IS
the probe — login, the T21 landed-tenant verify, and a guaranteed
logout on exit. So the probe is literally `with AcumaticaClient(inst):
pass`, and a tenant-guard refusal surfaces as `fail rest` with the
guard's own message. The ssh side got the one new primitive,
`TenantManager.ping()` — `Write-Output pong` through the `_ssh` choke
point, so V18's exit-code propagation covers it like every other remote
command. Both probes run through the exact objects live commands use;
a pass proves the real code path, not a parallel resolution.

Two small structural notes. The discovery probe needed the acu.yaml
parse without the credential merge, so `load_instance`'s parse step
became a shared `read_config()` instead of duplicated logic. And the
test for the .env-sourced password surfaced a pytest subtlety worth
keeping: `load_dotenv` mutates `os.environ` in place, and monkeypatch
only restores what it recorded — a setenv-then-delenv pair makes it
record the pre-test state so the .env value cannot leak into later
tests. `make check` green (139 tests).
